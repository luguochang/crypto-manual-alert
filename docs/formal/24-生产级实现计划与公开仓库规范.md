# 生产级实现计划与公开仓库规范

## 1. 结论

后续代码实现必须同时满足两类目标：

```text
真实可用
  -> 能在本地和个人服务器稳定运行，能手动 query、定时提醒、Bark 推送、Trace/Eval 回放。

公开可展示
  -> 可以推送到 GitHub 给面试官看，体现 TypeScript + Python 工程能力、架构能力、测试意识和安全边界。
```

本项目不是脚本堆叠，也不是为了炫技引入复杂中间件。实现标准是：

- 模块边界清楚。
- 类型和接口清楚。
- 配置集中管理。
- 错误和降级路径可观测。
- secret 不进入仓库。
- 关键业务函数有中文注释或 docstring。
- 核心 gates / workflow / eval 有测试。
- README、部署说明、架构图和演示路径完整。

## 2. 公开仓库安全规范

### 2.1 禁止提交

禁止进入公开仓库：

```text
真实 OPENAI_API_KEY
真实 OPENAI_BASE_URL 私有中转地址
真实 Bark key
真实交易所 API key / secret / passphrase
真实账户信息
真实持仓截图
真实交易记录原文
本地 .env
data/*.db
data/*.sqlite
logs/*
包含完整 raw prompt / completion 的 artifact
```

### 2.2 必须提供

必须提供示例和脱敏说明：

```text
.env.example
config/default.yaml
config/local.example.yaml
README.md
docs/formal/
```

`.env.example` 只保留字段名和说明：

```text
OPENAI_BASE_URL=https://example.com/v1
OPENAI_API_KEY=example-openai-api-key
OPENAI_MODEL=gpt-example
BARK_DEVICE_KEY=example
```

### 2.3 Secret 展示规则

前端、API、日志只能展示：

```json
{
  "configured": true,
  "masked": "sk-****abcd"
}
```

不能返回完整 secret。

### 2.4 提交前安全检查

每次推送前至少执行：

```text
检查 .env 是否被 git 跟踪
检查 data/*.db 是否被 git 跟踪
全文搜索 OPENAI_API_KEY / BARK / sk- / Authorization / passphrase
检查 README 中没有真实 key
检查 trace/eval 示例使用脱敏样本
```

后续可增加 pre-commit 或 CI secret scan，但首版先用文档和脚本约束。

## 3. 代码注释规范

### 3.1 总原则

注释用于解释业务意图、边界、风险和不显然的设计，不写机械废话。

应该写：

- 为什么这个 gate 必须 hard block。
- 为什么 search-derived 不能替代 exchange-native。
- 为什么 eval/replay 不能发 Bark。
- 某个状态为什么只允许从 A 到 B。
- 某个函数的输入输出契约。
- 某个配置项不能放宽的原因。

不应该写：

- “把 a 赋值给 b”。
- “循环列表”。
- “调用函数”。
- 和代码完全重复的描述。

### 3.2 Python 注释

公共类和公共函数使用中文 docstring：

```python
def build_decision_request(...):
    """将手动 query 或定时模板归一化为 DecisionRequest。

    该函数只负责构建请求语义，不拉行情、不调用 LLM、不做交易判断。
    """
```

复杂业务块可加短注释：

```python
# search-derived 证据只能作为上下文，不能满足开仓所需的核心执行事实。
```

必须加注释的模块：

```text
workflow/*
gates/*
skills/*
agents/*
eval/*
storage/*
api/schemas.py
```

### 3.3 TypeScript 注释

核心 API client、hook、复杂组件使用 TSDoc 或简明中文注释：

```ts
/**
 * 提交手动运行请求。
 * 返回 traceId 后，前端通过轮询或 SSE 读取运行进度。
 */
export async function createManualRun(...) {}
```

复杂 UI 状态需要注释：

```ts
// L3 配置只能创建候选版本，不能直接写入当前生产配置。
```

必须加注释的模块：

```text
frontend/src/lib/api/*
frontend/src/lib/schemas/*
frontend/src/features/runs/*
frontend/src/features/eval/*
frontend/src/features/candidates/*
```

## 4. 分层实现规范

### 4.1 Python 分层

Python 代码按下列方向重构：

```text
api/
  FastAPI 路由和请求/响应 schema，不写业务决策。

workflow/
  编排步骤、StepSpec、RunExecutor、状态流转。

context/
  DecisionRequest、Session、Memory、RunContext。

skills/
  skill 加载、SkillRegistry、ToolPolicy。

agents/
  intent、query rewrite、research planner、reviewer、final decision。

gates/
  FactsGate、ParserGate、PlanSemanticGate、RiskGate、SideEffectGate。

eval/
  FrozenInput、EvalCase、RuleJudge、LLMJudge、ReplayRunner、Report。

storage/
  SQLite/PostgreSQL repository，不让业务层直接写 SQL。
```

路由层禁止直接调用 LLM、OKX 或 Bark。路由只创建请求、调用 service/workflow、返回状态。

### 4.2 TypeScript 分层

前端按 feature 组织：

```text
frontend/src/
  app/
  components/
  features/
    dashboard/
    manual-run/
    runs/
    schedules/
    evidence/
    eval/
    candidates/
    settings/
  lib/
    api/
    schemas/
    formatters/
```

原则：

- 页面只组合组件，不写复杂业务逻辑。
- API 调用集中在 `lib/api`。
- Zod schema 集中在 `lib/schemas`。
- 表格、时间线、状态 badge 做成可复用组件。
- secret 只展示 masked 状态。

## 5. API 与类型规范

### 5.1 API 契约

所有 API 返回统一结构：

```json
{
  "ok": true,
  "data": {},
  "error": null,
  "trace_id": "optional"
}
```

失败返回：

```json
{
  "ok": false,
  "data": null,
  "error": {
    "code": "validation_error",
    "message": "请求参数不合法",
    "detail": {}
  },
  "trace_id": "optional"
}
```

错误码必须稳定，避免前端解析中文错误文本。

### 5.2 Schema 来源

首版允许 Python Pydantic 和 TypeScript Zod 分别维护，但字段名必须一致。

后续可做：

- FastAPI OpenAPI 导出。
- TypeScript API 类型生成。

首版不因类型生成引入额外复杂度。

## 6. 配置规范

配置分三类：

```text
环境变量
  -> secret 和部署差异。

YAML 配置
  -> 非 secret 的运行参数、定时任务、workflow 参数。

数据库配置
  -> 前端可改的 schedules、candidates、review 状态。
```

硬边界不允许配置关闭：

```text
manual-only
禁止自动交易
禁止下单/撤单/提现工具
search-derived 不可替代 mark/index/order_book
eval/replay 不发 Bark
RiskGate 不可关闭
ParserGate 不可关闭
```

## 9. 分阶段实现计划

### P0：公开仓库基线与安全清理

目标：

- 清理真实 secret。
- 完善 `.env.example`。
- 完善 `.gitignore`。
- 写 README 初版。
- 固化测试命令。

验收：

- 仓库可公开。
- secret scan 无明显命中。
- README 能让面试官理解项目。

### P1：Python FastAPI 最小 API

目标：

- `/api/system/health`
- `/api/runs/manual`
- `/api/runs`
- `/api/runs/{trace_id}`

验收：

- 前端可创建 run。
- run 可落库。
- trace 可查询。

### P2：Next.js 工作台基础

目标：

- Dashboard。
- Manual Run。
- Runs / Trace。
- API client。
- 基础 layout。

验收：

- 本地启动后能从 UI 提交 query。
- 能看到运行状态和 trace 时间线。

### P3：DecisionRequest / Workflow / Gates 重构

目标：

- 按 22 号文档引入 DecisionRequest。
- 引入 FactsGate、PlanSemanticGate、SideEffectGate。
- 保留 legacy facade。

验收：

- 旧 CLI 不破。
- 新 API 能走统一 workflow。
- 核心 gates 有测试。

### P4：Schedules 与 Bark

目标：

- 定时 query CRUD。
- Python scheduler loop。
- Bark 通知记录。
- 前端 Schedules 页面。

验收：

- 可配置定时任务。
- 可触发 Bark。
- Bark 失败不改变决策结果。

### P5：Eval / Cases / Candidates

目标：

- EvalCase。
- RuleJudge。
- Candidates。
- 前端 Eval/Candidates 页面。

验收：

- eval/replay 不发 Bark。
- candidate 不能直接生效。
- 能看到 baseline/candidate 结果。

### P6：完善展示与工程质量

目标：

- README 完整。
- 示例截图或流程图。
- 测试覆盖补齐。
- 类型检查、lint、build。
- 文档与代码一致。

验收：

- 适合推送到公开 GitHub。
- 面试官能从 README、代码结构、测试和 UI 看懂项目价值。

## 11. 代码评审清单

每次合并前检查：

- 是否引入了额外中间件。
- 是否绕过了 Python hard gates。
- 是否把 secret 返回给前端。
- 是否把 eval/replay 接到了 Bark。
- 是否把历史 badcase 注入 live prompt。
- 是否新增了缺注释的复杂业务函数。
- 是否缺少测试。
- 是否破坏旧 CLI。
- 是否改变公开 API 却没更新文档。
- 是否引入真实数据或真实 key 到仓库。

# 通知与调度策略

## 通知目标

通知用于提醒用户手动复核和手动操作。

系统不自动下单。

## 通知渠道

v1 必须支持：

- Bark

预留：

- 飞书

## 通知内容

必须包含：

- main action
- instrument
- horizon
- entry trigger
- stop
- target 1 / target 2
- probability
- allowed / blocked 状态
- confidence cap reason
- why not opposite 简述
- unavailable data
- expiry
- 手动操作提示

必须包含固定提示：

```text
手动操作必需，系统没有下单。
```

## 通知降噪

定时任务不应每轮无差别刷屏。

建议 v1 规则：

- 手动 query 每次都通知。
- 定时 query 只在以下情况通知：
  - main action 变化
  - allowed / blocked 状态变化
  - probability 跨过配置阈值
  - 出现 hard block
  - LLM / 数据源连续失败

每次运行仍必须写 journal。

## Blocked 计划是否通知

建议：

- 手动 query：通知 blocked，告诉用户不能直接执行。
- 定时 query：只在首次 blocked 或状态变化时通知。

## 通知失败

通知失败不能改变 `RiskVerdict`。

处理：

```text
记录 notification failure
不重跑决策
不改变 journal 中的 plan verdict
```

## 调度策略

### 频率

建议默认：

```yaml
scheduler:
  interval_seconds: 1800
  job_timeout_seconds: 1800
```

不建议低于 5 分钟。

原因：

- skill 分析耗时较长。
- LLM 调用成本和延迟较高。
- 市场噪声过高会导致频繁提醒。

### 耗时边界

单轮任务不是 60 秒任务。Leader 规划、并发检索、Leader 对抗审查、最终决策串起来时，10 到十几分钟属于可接受范围。

超时语义必须拆开：

- `research.request_timeout_seconds`：单个 research planner / researcher search / leader reviewer 请求的超时，默认建议 300 秒。
- `decision.timeout_seconds`：最终决策模型单次请求超时，默认建议 900 到 1200 秒。
- `scheduler.job_timeout_seconds`：单轮任务预算或外部 watchdog 参考值，默认建议 1800 秒。

不要用 60 秒作为整轮任务硬限制。真正需要限制的是单个外部请求不能无限挂起，以及 scheduler 不能并发重入。

### 锁

每个 job 必须有锁。

拿不到锁：

```text
skip current run
write lightweight log
do not run in parallel
```

### 异常

单个 job 失败不能打断 scheduler 循环。

失败必须：

- 写 journal 或 failure log。
- 按配置发送 failure notification。

## 定时任务配置示例

```yaml
scheduled_queries:
  - name: eth_6h
    enabled: true
    symbol: ETH-USDT-SWAP
    horizon: 6h
    position_state: long
    interval_seconds: 1800
    notify_on:
      action_change: true
      status_change: true
      hard_block: true
      every_run: false
```

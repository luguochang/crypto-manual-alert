# 业务层重构整改方案与架构设计

## 1. 结论

这次业务层重构的目标不是“把项目改得更像 agent 平台”，而是把交易提醒业务从当前的：

```text
symbol -> 拉行情 -> 可选 research -> LLM 输出 JSON -> parser -> risk strings -> Bark
```

升级为：

```text
DecisionRequest
  -> Facts / Evidence
  -> FactsGate / DataQualityGate
  -> Research Coverage
  -> Review Contributions
  -> DecisionInputBuilder
  -> FinalDecision
  -> ParserGate
  -> PlanSemanticGate
  -> RiskGate
  -> Trace / FrozenInput / Eval
```

核心判断：

- 当前真正影响效果的缺陷不是观测字段少，而是业务输入、事实证据、反向审查和计划语义没有形成硬契约。
- 业务层要先补 `DecisionRequest`、`EvidencePacket`、`FactsGate`、`DecisionInputBuilder`、`PlanSemanticGate`、结构化 `RuleHit`。
- `StepSpec`、query 级 span、FrozenInput、Eval 是必要基础，但它们本身不直接提升交易判断质量，不能被包装成效果提升。
- 独立多 reviewer、LLMJudge、Streamlit、Langfuse/Phoenix 只能后置；先做会制造复杂度和假精确。

一句话：

```text
目标架构一次性想清楚；实现必须渐进、可回滚、可对比旧链路，任何阶段都不能降低现有手动提醒安全性。
```

## 2. 为什么之前没有一次考虑到位

需要直接承认：前面的设计偏 MVP 链路，低估了后续“持续优化、复盘、测评、长期维护”的复杂度。

当时的设计重点是：

- 先跑通个人使用的手动提醒闭环。
- 禁止自动交易。
- 接入 Bark。
- 用 `crypto-macro-decision` skill 作为规则来源。
- 用 Parser/RiskGate 做 fail closed。
- 能本地定时跑起来。

这些优先级让系统很快形成了一个可用 pipeline，但产生了结构性欠账：

| 欠账 | 当时被低估的原因 | 现在的影响 |
|---|---|---|
| 只有 `symbol`，没有 `DecisionRequest` | 以为固定 ETH/BTC/SOL 手动提醒足够 | 持仓、周期、用户意图不能稳定进入决策 |
| facts 和 evidence 混在 snapshot | 以为 RiskGate 能兜住核心行情缺失 | LLM prompt 中上下文和执行事实边界不清 |
| research 只做 fallback | 以为 OKX 完整时不需要外部研究 | 宏观、BTC 锚、拥挤度、事件窗口可能被漏掉 |
| 多 agent 只是 prompt 结构化角色 | 以为多角色输出足够形成对抗 | reviewer 没有独立耗时、失败、证据引用 |
| trace 后补 | 以为先跑通后再观测 | 后续 badcase 无法高质量 replay |
| 没有 FrozenInput | 以为 journal payload 足够复盘 | 不能做 baseline/candidate 对比 |
| Eval 没有先作为开发门禁 | 以为测试用例能覆盖主要风险 | prompt/research/risk 改动无法证明增益 |

这个复盘不是为了否定当前系统。当前系统已经有 Parser/RiskGate、trace、badcase、Bark 和基础 research，这些是有价值的。问题是：它还不是一个可持续演进的 AI 应用架构。

后续必须把“可复盘、可测评、可回滚”作为开发前置条件，而不是用户指出问题后再补。

## 3. 重构成功标准

成功标准不能是“目录更漂亮”或“agent 更多”，必须能证明以下结果：

| 目标 | 验收方式 |
|---|---|
| 不降低安全性 | 同一输入下，新链路不得允许旧链路阻断的高风险开仓 |
| 不降低可用性 | 旧 CLI/scheduler/Bark 入口不中断；默认仍能跑 fixture 和生产配置 |
| 提升业务语义 | trace 中能看到 request、position、horizon、facts gate、evidence refs、rule hits |
| 提升证据质量 | 每条证据有来源、时间、新鲜度、质量、是否可满足执行事实 |
| 提升反向审查 | 反向链从自由文本升级为结构化 `CounterThesis`，空泛反证可触发降级 |
| 提升计划可靠性 | 增加价位关系、action-position、TTL、risk/reward 等语义 gate |
| 支持后续 Eval | 可从任意生产 trace 生成 FrozenInput 和 EvalCase |
| 可回滚 | 每阶段都有 feature flag、legacy facade 或 shadow 模式 |

明确不作为首阶段成功标准：

- 不承诺提高收益率。
- 不承诺 LLMJudge 立刻可作为 hard gate。
- 不承诺独立 reviewer 一定比 compact reviewer 更准。
- 不承诺首版 UI 或第三方平台带来质量提升。

## 4. 当前业务层缺陷优先级

### 4.1 P0 效果缺陷

这些直接影响交易提醒质量。

1. 缺少 `DecisionRequest`
   - 当前入口只有 `symbol`。
   - 但输出动作包含 `hold/close/flip/open/trigger`。
   - 没有持仓状态时，系统不应该轻易输出 `hold/close/flip`。

2. 事实层和证据层没有分离
   - `MarketSnapshot.points` 同时承载 OKX 原生数据和 `web_*`。
   - 缺少 `source_type/quality/freshness/can_satisfy_execution_fact`。
   - 搜索上下文可能在 prompt 中被误读为执行事实。

3. research 只是行情失败后的 fallback
   - OKX 数据完整时可能跳过 BTC 方向锚、宏观、ETF、清算、拥挤度、重大事件窗口。
   - 对 crypto macro 决策来说，这会漏掉主要驱动。

4. 反向链没有成为 gate 输入
   - `why_not_opposite` 主要依赖 prompt 要求。
   - Parser/RiskGate 不校验其质量。
   - 结果可能出现“有字段但内容空泛”。

5. 缺少 `PlanSemanticGate`
   - 当前 parser/risk 还没有校验多空价位关系、target 顺序、action 与持仓兼容、TTL 与周期匹配、risk/reward。

### 4.2 P1 治理缺陷

这些不一定直接提升效果，但影响持续优化。

1. `research.search` 不是 query 级 span。
2. RiskGate 只有中文 reason，没有稳定 `rule_id`。
3. LLM interaction 缺 token/cost/finish_reason/prompt_version。
4. `raw_decision` 仍在生产 payload。
5. 没有 FrozenInput 和 replay。
6. 没有 release gate。

## 5. 目标业务架构

### 5.1 分层原则

按业务语义分层，不按“agent 数量”分层。

```text
request 层
  -> facts 层
  -> gates 层
  -> research 层
  -> review 层
  -> decision 层
  -> runtime 层
  -> trace/eval 层
```

### 5.3 模块边界

| 层 | 负责 | 不负责 |
|---|---|---|
| `context` | 请求、运行上下文、持仓/周期/用户意图 | 拉行情、调用 LLM |
| `models` | Evidence、RuleHit、Review、FrozenInput 等结构化对象 | 执行业务流程 |
| `gates` | 确定性规则、硬边界、语义校验 | 市场方向判断 |
| `workflow` | 步骤编排、timeout、retry、fallback、上下文流转 | 交易观点 |
| `tools` | OKX、web search、LLM、Bark IO | 写业务结论 |
| `skills` | skill 加载、受控工具注册、ToolPolicy | 自动发现任意 skill |
| `agents` | 研究规划、review、final decision 的结构化产出 | 修改 RiskVerdict、发通知 |
| `rules` | 可收紧规则和 prompt 配置加载 | 关闭安全硬边界 |
| `eval` | FrozenInput、EvalCase、RuleJudge、Replay | 影响 live decision |

## 6. 核心对象设计

### 6.1 DecisionRequest

```text
request_id
run_type: manual | scheduled | replay | eval
symbol
horizon
position:
  side: long | short | flat | unknown
  entry_price
  size
  leverage
user_query_summary
created_at
source
```

效果收益：

- 已持多单时，优先允许 `hold long/close long/flip long to short`。
- 空仓时，不应输出 `hold/close/flip`。
- 持仓未知时，`hold/close/flip` 至少触发 warning 或 block。
- 周期明确后，TTL、目标、止损距离可以做语义检查。

### 6.2 EvidencePacket

```text
evidence_id
name
value
source_type: exchange_native | skill_script | responses_web_search | duckduckgo_html | fixture
source_url
published_at
fetched_at
freshness_seconds
quality: primary | secondary | search_derived | inferred
status: ok | stale | unavailable | conflicting
can_satisfy_execution_fact
claims
conflicts
confidence_cap
unavailable_reason
```

硬边界：

```text
mark/index/order_book 只能由 exchange_native 或明确白名单 skill_script 满足。
search_derived 永远不能满足核心执行事实。
```

### 6.3 FactsGateResult

```text
passed
missing_core_points
stale_points
execution_points_available
research_required
confidence_caps
hard_blocks
soft_warnings
input_refs
```

用途：

- 替代“从是否出现 research span 推断缺口”。
- 作为 final decision 输入的一部分。
- 作为 RuleJudge 和 FrozenInput 的稳定字段。

### 6.4 CounterThesis

```text
side_against_main_action
strongest_opposite_chain
supporting_evidence_refs
confirmation_trigger
invalidation_trigger
why_rejected
confidence_effect
quality: sufficient | weak | missing
```

效果收益：

- 反向链不再只是 `why_not_opposite` 自由文本。
- 空泛反证可以触发 confidence cap、advisory warning 或 `no trade`。

### 6.5 RuleHit

```text
rule_id
severity: info | warning | hard_block
category
message
input_ref
threshold
actual
passed
```

兼容：

- `RiskVerdict.reasons` 保留。
- 新增 `rule_hits`，逐步让 eval 使用 `rule_id`，而不是解析中文 reason。

### 6.6 PlanSemanticGate

负责 ParserGate 和 RiskGate 之间的交易计划语义校验。

首批规则：

| rule_id | 规则 |
|---|---|
| `plan.price.long_stop_below_entry` | 多头开仓/触发时 stop 必须低于 entry |
| `plan.price.short_stop_above_entry` | 空头开仓/触发时 stop 必须高于 entry |
| `plan.price.target_order` | target_1/target_2 与方向一致 |
| `plan.position.action_compatible` | action 必须与 position_state 兼容 |
| `plan.ttl.horizon_compatible` | `expires_in_seconds` 与 horizon 不冲突 |
| `plan.rr.minimum` | 风险收益比低于阈值时 warning 或 block |
| `plan.mark.deviation` | entry/reference 与 mark 偏离过大时 warning |

上线策略：

- 价格方向明显错误先 hard block。
- risk/reward、mark deviation 初期 advisory。
- 避免一次性把可用性打死。

## 7. 快速提醒链路与深度研究链路

必须区分两条链路，避免“更完整”变成“过时”。

### 7.1 快速提醒链路

用途：

- 定时巡检。
- 用户要“现在怎么操作”。
- 优先输出安全、清晰、可手动执行的计划。

默认上限：

```text
total_timeout: 3-5 分钟
market_fetch: 25s
research: 0-180s
decision: 300-600s
```

策略：

- OKX 核心事实优先。
- research coverage 使用轻量查询。
- compact reviewer。
- 缺关键事实就触发 `trigger/no trade/blocked`，不拖无限研究。

### 7.2 深度研究链路

用途：

- 用户明确要求多 agent 根因链。
- 复盘。
- 生成 eval case。
- 高风险重大事件前审查。

默认上限：

```text
total_timeout: 10-20 分钟
research_total: 600s
review_total: 600s
decision_total: 1200s
```

策略：

- 更完整 research coverage。
- 可启用 independent reviewers。
- 更重的 LLMJudge 或人工复核。
- 输出可以更慢，但必须标记运行模式。

## 8. 直接提升效果的改造

这些改造预期会直接提升交易提醒质量。

| 改造 | 为什么提升效果 | 验收 |
|---|---|---|
| `DecisionRequest` | 避免持仓未知时错误 hold/close/flip | action-position 兼容测试 |
| `EvidencePacket` | 防止上下文被当作执行事实 | search-derived 永不满足 execution fact |
| `FactsGate` | 明确数据缺口、stale、cap | trace 中有 gate result |
| research coverage | OKX 完整时也检查 BTC 锚、宏观、拥挤度、事件 | coverage report 包含必需域 |
| source quality | 搜索结果必须有时间、来源等级、claim | 无时间来源降权或 cap |
| `CounterThesis` | 反向链从文本变成结构化输入 | 空泛反向链触发 warning/cap |
| `DecisionInputBuilder` | final LLM 只消费受控事实、证据、hard block | prompt 不再直接塞大 snapshot |
| `PlanSemanticGate` | 校验价格、方向、持仓、TTL、RR | 明显错误计划 hard block |

## 9. 服务 Eval/维护的改造

这些是必要基础，但不能夸大为直接提升效果。

| 改造 | 价值 | 不应夸大的点 |
|---|---|---|
| query 级 span | 定位哪个查询慢/错 | 不等于查询质量变好 |
| token/cost/latency | 控制成本和时效 | 不等于判断更准 |
| FrozenInput | 支持 replay 和 baseline/candidate | 不改变 live 输出 |
| EvalCase / RuleJudge | 防回归 | 不直接预测市场 |
| LLMJudge | 评估 grounding/过度自信 | 不能评收益，不能替代 RiskGate |
| Streamlit UI | 降低人工复核成本 | UI 不代表数据可靠 |
| 第三方 exporter | 便于外部观测 | 不作为首版强依赖 |
| YAML prompts/rules | 版本管理和可维护 | 安全硬边界不能被配置关闭 |

## 10. 迁移阶段

### P0：冻结契约，补业务语义

目标：

- 不改变外部入口。
- 先把业务请求和结果契约补起来。

新增：

```text
src/crypto_manual_alert/context/request.py
src/crypto_manual_alert/context/run_context.py
src/crypto_manual_alert/models/rule_hit.py
src/crypto_manual_alert/models/evidence.py
```

修改：

- `PlanRunner.run_once(symbol)` 内部构造默认 `DecisionRequest`。
- scheduler 调用时写 `run_type=scheduled`。
- `RiskVerdict` 增加兼容字段 `rule_hits=[]`，保留 `reasons`。
- `why_not_opposite` 和 `invalidation` 从“有字段”升级到“非空最低要求”。

测试：

```text
tests/test_decision_request_context.py
tests/test_runner_legacy_contract.py
```

验收：

- `PlanRunner.run_once(symbol)` 返回类型不变。
- CLI 输出不变。
- trace 中出现 request 摘要。
- 旧测试全通过。

回滚：

- 关闭新 request metadata 写入。
- 继续走旧 `symbol` 参数。

### P1：抽 Gates，不改外部行为

目标：

- 把 facts/parser/risk 变成显式 gate。
- 让 gate 可被 eval 使用。

新增：

```text
src/crypto_manual_alert/gates/facts_gate.py
src/crypto_manual_alert/gates/parser_gate.py
src/crypto_manual_alert/gates/risk_gate.py
src/crypto_manual_alert/gates/plan_semantic_gate.py
```

修改：

- `runner.py` 从直接调用 `needs_research_fallback()` 改为调用 `FactsGate` wrapper。
- `risk.py` 输出稳定 `rule_id`。
- `parser_gate.py` 包装现有 `parse_decision_plan()`。
- `PlanSemanticGate` 初期 shadow/advisory。

测试：

```text
tests/test_gates.py
tests/test_plan_semantic_gate.py
```

验收：

- `facts.gate` span 出现在 trace。
- 缺核心行情、陈旧行情、search-derived 边界有 rule_id。
- Parser/Risk 旧行为不变。

回滚：

- wrapper 回退到旧函数调用。
- `PlanSemanticGate` 保持 advisory。

### P2：建立 DecisionWorkflow，PlanRunner 变 facade

### P3：重构 Research，但保持 compact reviewer

### P4：启用 DecisionInputBuilder + PlanSemanticGate

目标：

- final LLM 不再直接面对混杂大 payload。
- 输出前后都有语义约束。

新增：

```text

src/crypto_manual_alert/models/decision_input.py
```

修改：

- `skill_runtime.build_prompt_packet()` 逐步变成 `DecisionInputBuilder`。
- prompt 包含 facts、gaps、evidence_refs、review_contributions、allowed_actions、hard_blocks。
- `PlanSemanticGate` 将高置信规则切 hard。

测试：


- 明显错误价位关系被 block。
- 持仓未知时 `hold/close/flip` 被 warning 或 block。

回滚：

- prompt_version 回退旧版。
- `PlanSemanticGate` 降回 advisory。

## 11. 不降低效果的保护机制

### 11.1 Shadow Migration

新链路初期只旁路记录，不负责生产 Bark。

```text
旧 pipeline -> 生产输出
新 workflow shadow -> trace / FrozenInput / RuleJudge
```

只有满足 gate 后，才逐步让新 workflow 接管对应步骤。

### 11.2 行为等价检查

同一 fixture / frozen input 下：

- parsed plan 应一致，或差异有 ADR。
- RiskVerdict 应一致，或新链路更保守且有 rule_id。
- 新链路不得允许旧链路阻断的 opening action。
- `no trade` 比例不能无解释显著上升。

### 11.3 Feature Flags

建议开关：

```yaml
workflow:
  engine: legacy | new_shadow | new_active

gates:
  facts_gate_mode: shadow | advisory | hard
  plan_semantic_gate_mode: shadow | advisory | hard

research:
  coverage_mode: fallback_only | lightweight_always | deep

agents:
  reviewer_mode: compact | independent

decision:
  input_builder_version: legacy | structured_v1
```

### 11.4 快速/深度分离

快速提醒默认不启用重型 independent reviewer。

深度研究可以跑更久，但必须标记 `run_mode=deep_research`，避免用户误以为这是实时短线提醒。

## 14. 多 Agent 对抗审查采纳

本轮启动三个只读审查 agent：

| Agent | 角色 | 核心结论 | 采纳 |
|---|---|---|---|
| 业务架构审查 | 从效果提升排序缺陷 | 最大缺陷是缺 `DecisionRequest`、事实/证据分离、research coverage、反向链和计划语义 gate | 采纳到第 4、6、8 节 |
| 红队审查 | 防止再次过度设计 | 不要把架构先进作为成功标准；必须 shadow migration；不要过早上 independent reviewer/LLMJudge/UI | 采纳到第 1、3、11、12 节 |
| 渐进迁移审查 | 工程迁移计划 | 旧入口保留，新目录渐进迁移；每阶段有测试、兼容、回滚和记录 | 采纳到第 5、10、13 节 |

对抗后形成的关键修正：

- `DecisionRequest` 是第一优先级，不是 Eval 或 UI。
- research 要从“fallback”升级为“coverage”，但快速链路和深度链路必须分开。
- `PlanSemanticGate` 是业务效果改造，优先级高于独立多 agent。
- independent reviewer 只在有证据证明 compact 不够时启用。
- LLMJudge 只能后置 advisory，不作为首阶段 gate。
- 每轮开发必须记录迁移文档和 ADR，避免后续无法复盘。

## 15. 不做事项

当前不做：

- 不自动交易。
- 不注册下单、撤单、提现工具。
- 不让 eval 发送 Bark。
- 不让 eval 写生产 journal。
- 不把 badcase 注入 live prompt。
- 不用历史结论污染当前 live decision。
- 不把所有规则都 YAML 化。
- 不用 LLMJudge 替代 RiskGate。
- 不直接引入完整 LangGraph/AutoGen/CrewAI 接管主流程。
- 不把 UI 或第三方平台作为第一阶段目标。

## 16. 最终建议

下一步应该做 P0/P1，不要直接进入大规模 agent 重写：

```text
P0:
  DecisionRequest
  EvidencePacket 基础模型
  RuleHit 兼容字段
  request trace metadata
  baseline contract tests

P1:
  FactsGate
  ParserGate wrapper
  RiskGate structured rule hits
  PlanSemanticGate shadow
```

只有这两步完成，后续 eval 才有真实业务语义可以评。否则继续堆 LLMJudge、UI、独立 agent，只会让系统看起来更成熟，但结论仍然不可复盘、不可证明。

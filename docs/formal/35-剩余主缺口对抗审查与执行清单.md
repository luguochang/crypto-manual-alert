# 剩余主缺口对抗审查与执行清单

> **当前定位（2026-07-09）**：本文是历史 AgentSwarm 缺口清单和对抗审查记录，不是当前日常执行入口。当前交付执行、P0 证据闭环和 checklist 以 `docs/implementation/2026-07-09-current-delivery-checklist.md` 为准。文中已完成项多为字段/投影/审计侧路完成，不代表生产候选 AgentSwarm 已接管主链。

## 1. 文档定位

本文是 `34-生产级AgentSwarm优化目标与执行计划.md` 之后的剩余主缺口收敛计划。

当前已经完成的关键基础：

- `llm_tool_shadow` 在 fixture 模式下可通过主 API 链路产出真实 `ToolCallArtifact`。
- `skills/` 已按业务能力拆为 `realtime_search/`、`root_cause/`、`sentiment_crowding/`、`macro_event/`、`liquidity_order_book/`。
- `LiveFactAgent` 已进入受控 `realtime_search` skill 链路。
- `root_cause_search` 已有受控递归 provider 边界。
- `liquidity_order_book` 已有 `fact_refs`，可投影 `mark/index/order_book` refs。
- `production_candidate_swarm` 执行模式已存在，但仍是 blocked audit-only。
- `production_candidate_swarm` 已运行 audit-only candidate FinalDecisionAgent sidecar。
- Run Detail 页面已显示 `Fact Refs`，并通过 runtime page smoke。

本文要解决的问题：

- 防止后续再次偏到“文档很多、结构测试很多、主链路没有实际提升”。
- 把剩余工作拆成可验收 checkpoint。
- 每个 checkpoint 必须说明改哪些文件、怎么验收、何时暂停。

## 2. 对抗审查说明

当前会话没有可调用的独立 subagent 调度工具，因此本轮采用多角色对抗审查方式记录结论。后续如果有真实 subagent 工具，应按本文角色分别派发只读审查。

### 2.1 架构审查角色

结论：

- 当前 `production_candidate_swarm` 只是安全入口，不是真正生产候选主链。
- 如果下一步直接接真实 provider，但没有 gate/release 边界，会放大风险。
- 必须先定义 `audit_only`、`production_candidate`、`blocked` 的状态切换条件。

必须补齐：

- candidate/legacy/gate 三方对比。
- production candidate readiness gate。
- release review 前不得切默认 final input。

### 2.2 后端主链路审查角色

结论：

- `/api/runs/manual -> RunExecutor -> adapter -> journal -> agent_audit_view -> frontend` 已能跑通。
- 但 legacy baseline 和 production candidate swarm 仍是两条不完整链路。
- `controlled_adapter` 当前没有完整 market/research/legacy final 对比。

必须补齐：

- `production_candidate_swarm` 的候选链路状态必须更细。
- candidate sidecar 的 gate failure 必须变成可读诊断。
- 不能把 blocked no-trade 误当成真实候选交易计划。

### 2.3 Skill/Provider 审查角色

结论：

- Skill 包结构已经改善，但 provider 多数仍是 fixture 或注入边界。
- `realtime_search`、`root_cause_search`、`liquidity_order_book` 还没有真实 provider 闭环。
- provider 一旦真实接入，timeout、budget、freshness、source tier 必须贯穿。

必须补齐：

- realtime web search provider。
- root-cause 递归使用真实 search provider。
- exchange-native order book adapter。
- provider 错误 envelope 和降级策略。

### 2.4 前端可观测审查角色

结论：

- 页面已经能显示 Fact Refs，但整体仍偏日志浏览器。
- 管理者需要首屏看到“能不能信、为什么不能执行、缺什么”。
- JSON 只能作为辅助，不能作为主表达。

必须补齐：

- 首屏风险摘要。
- Tool/Worker/Gate/Input Lineage 的清晰链路。
- candidate failure reason 的中文可读摘要。
- financial quality not configured 的显式提示。

### 2.5 Eval/金融质量审查角色

结论：

- 结构安全 gate 已经有雏形，但金融质量闭环仍不完整。
- 没有 outcome collector，无法证明预测是否有效。
- release gate 不能只看 schema/安全，还要看样本质量。

必须补齐：

- outcome collector。
- financial metrics。
- baseline 对比。
- eval 页面展示。

### 2.6 安全/发布审查角色

结论：

- 当前没有自动下单风险，但真实 provider 和生产候选模式会提高误用风险。
- 必须维持默认保守配置。
- 所有“生产候选”必须通过人工 release review。

必须补齐：

- policy 外显。
- provider enable 开关。
- release review checklist。
- 明确回滚路径。

## 3. 全局不变约束

- 不默认设置 `decision.final_input_mode=decision_input`。
- 不默认设置 `workflow.execution_mode=production_candidate_swarm`。
- 不默认设置 `shadow.worker_mode=llm_tool_shadow`。
- 不写订单、不撤单、不发交易通知。
- `manual_execution_required=true` 必须保持。
- FinalDecisionAgent 不得调用 tool。
- Worker 不得输出最终交易动作字段。
- search-derived evidence 不得满足 mark/index/order_book execution facts。
- 同一问题连续失败 3 次必须暂停，记录证据和下一步选择。

## 6. 暂停条件

出现以下任一情况必须暂停，不继续硬改：

- 同一问题连续失败 3 次。
- 为了让测试通过需要关闭 gate。
- 需要默认启用真实外部 provider。
- 需要默认切换 final input。
- 页面 smoke 不能证明真实 trace 数据可见。
- provider 需要真实 key 但没有明确配置和脱敏策略。
- 代码改动开始跨越 3 个以上无关子系统。

## 7. 当前推荐执行顺序

优先级从高到低：

1. 阶段一：补完整 production candidate swarm 状态机。
2. 阶段二：真实 provider 边界。
3. 阶段三：前端驾驶舱。
4. 阶段四：金融质量闭环。
5. 阶段五：Harness policy 外显。
6. 阶段六：结构收敛。

不建议先做：

- 大规模目录搬家。
- 全量测试矩阵扩张。
- 长期记忆或向量库。
- 自动策略调参。
- 默认接真实外部 LLM/web/exchange。

## 8. 完成定义

只有同时满足以下条件，才能称为生产候选 Agent Swarm 基本完成：

- 一个真实 manual run 页面能看到 request、worker、skill、fact refs、candidate sidecar、gate、final input selection。
- 默认生产 final input 仍安全保守。
- 真实 provider 必须显式配置。
- execution facts 只来自 exchange-native refs。
- candidate final 与 legacy final 可比较。
- financial quality eval 能从历史 run 生成 outcome。
- release review 有结构安全和金融质量两类证据。
- 所有关键变更有 migration 记录和聚焦测试。

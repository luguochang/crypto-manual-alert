# 实施计划

## 原则

先收紧现有单线流程，再逐步加入 skill 固定加载、证据快照、根因链和通知优化。

不要一开始实现完整 swarm。

## Phase 1: 收紧现有运行边界

目标：

- 让当前单线流程 fail closed。

任务：

- 给 `PlanRunner.run_once()` 增加总异常边界。
- 失败也写 journal。
- scheduler 捕获 job 异常，不打断循环。
- 通知失败只记录，不改变 verdict。
- 移除或禁用生产 `CommandDecisionEngine shell=True` 路径。

测试：

- runner 异常。
- scheduler 异常。
- notifier 异常。
- journal 失败。

## Phase 2: 严格 JSON 和风控

目标：

- 不再宽松修复模型输出。

任务：

- 收紧 `parse_decision_plan()`。
- 禁止默认补字段。
- 增加 confidence cap。
- 增加 hard block / soft downgrade。

测试：

- 非法 JSON。
- fenced JSON。
- 字段缺失。
- probability 超 cap。
- stop 缺失。

## Phase 3: 固定 Skill 加载

目标：

- v1 明确以 `crypto-macro-decision` 为规则来源。

任务：

- 增强 `SkillRuntime`。
- 读取 `SKILL.md`。
- 读取必要 references。
- 计算 skill hash。
- 校验 `scripts/okx_snapshot.py`。
- journal 记录 skill context。

测试：

- skill 缺失。
- reference 缺失。
- hash 变化。

## Phase 4: EvidenceSnapshot

目标：

- 把市场事实和数据缺口结构化。

任务：

- 新增 `EvidenceSnapshot`。
- 包装调用 `okx_snapshot.py`。
- 转换 OKX 返回为统一 facts。
- 标记 unavailable / stale / source type。

测试：

- OKX 正常。
- OKX 部分失败。
- OKX 超时。
- 核心事实缺失。

## Phase 5: 根因链和执行风险审查

目标：

- 不做复杂 swarm，先做固定模板。

任务：

- 新增 `RootCauseReview`。
- 新增 `ExecutionRiskReview`。
- prompt packet 加入固定模板。
- finalizer 必须输出主链和反链摘要。

测试：

- 无反链时 block 或 retry。
- 无 invalidation 时 block。
- execution risk hard block。

## Phase 6: 通知和调度优化

目标：

- 减少定时任务刷屏。

任务：

- 支持 notify_on 配置。
- blocked 手动 query 通知。
- 定时任务只在状态变化通知。
- 飞书预留接口。

测试：

- action change。
- status change。
- hard block。
- every_run=false。

## Phase 7: 文档和部署验收

目标：

- 本地和服务器可按文档部署。

任务：

- 更新 quickstart。
- 更新 deployment。
- 更新 configuration。
- 更新 operation。
- 补服务器部署检查表。

验收：

- pytest。
- local run-once。
- Bark test。
- docker compose config。
- server dry run。

## 暂不实现

- 自动交易。
- read-only account position。
- 完整多 agent swarm。
- 动态 skill registry。
- 长期向量记忆。
- 回测系统。

# 架构功能收敛总计划与追踪清单

## 1. 文档定位

本文用于把后续重构目标、执行顺序、检查点和验收口径提前写清楚，避免再次出现“实现很多、但主链价值没有提升、代码越来越散”的问题。

权威关系如下：

- `31-受控AgentSwarm主链收敛与质量切换计划.md`：唯一进度勾选入口。执行过程中实际完成状态以 31 为准。
- 本文：总计划、架构边界、功能目标和详细追踪清单。后续如果执行方向变化，先更新本文的设计口径，再更新 31 的执行状态。
- `29-Agent与Skill拆分详细设计.md`：目标架构参考。
- 30 号实施契约文档：当前代码事实、兼容路径和边界契约参考，不再继续堆执行流水。

本文不使用版本号或优先级缩写作为后续追踪口径。后续统一使用“阶段、检查点、小闭环”描述。
本文不维护完成状态；所有实际勾选、完成记录、失败记录和迁移记录都必须回写 31。

## 2. 当前真实状态

当前项目已经完成一部分结构收敛，但主链还没有真正切到受控 Agent Swarm。

当前真实链路仍是：

```text
RunExecutor
  -> DecisionRunContext
  -> legacy adapter
  -> legacy decision workflow
  -> fixed pipeline / concurrent search / single leader review
  -> pre-final shadow audit
  -> final decision
  -> parser / production control gate / risk
  -> journal / notification
```

当前已经具备的能力：

- 根包业务 `.py` 基本清零，根包只保留包入口。
- `market_agents/` 已成为业务 Worker Agent 的主要归属目录。
- `agent_swarm/` 已收敛为 runtime、registry、compat、tool worker 等编排基础设施。
- `orchestration/` 承载 harness、contracts、shadow audit、failure envelope 等跨 Agent 编排契约。
- `decision/` 承载 DecisionInput、candidate final、switch readiness、生产控制 gate 和 replay 输入。
- `skills/facade.py` 已形成受控 skill facade，Skill 只返回结构化工具结果，不返回 `AgentContribution`。
- 现有 shadow/candidate/eval/replay 仍保持 sidecar/audit 语义，不允许生产副作用。

当前没有完成的关键点：

- 还没有把 `LeadAgent -> 多 Worker Agent -> SharedContext/ArtifactStore -> Lead synthesis -> DecisionInput` 变成主候选链路。
- 候选 FinalDecisionAgent 还没有被限制为只消费 `DecisionInput`。
- Release gate 还没有成为可证明、可回放、可人工审查的质量切换门禁。
- 生产默认仍必须是 `legacy_baseline + legacy_prompt`，不能打开 `decision.final_input_mode=decision_input`。
- 兼容 wrapper、旧测试路径、历史文档仍很多，需要在后续收口阶段处理生命周期，不能边实现边无限堆叠。

当前确认的问题记录：

- 问题一：当前代码已经拆出 7 个 Worker Agent 和 5 个 Skill facade，但默认真实主链仍是 `legacy_baseline + legacy_prompt`。Agent Swarm 目前主要作为 pre-final shadow audit / candidate sidecar 存在，`decision_effect=none`，还没有成为最终交易判断的主编排链路。
- 问题二：当前已有 trace、span、LLM interaction、plan payload、badcase 等后端记录能力，但页面仍主要展示运行级时间线和 LLM 请求/返回，没有完整展开 `LeadPlan`、每个 Worker Agent 的输入/输出、Skill 调用结果、Harness 校验、Lead synthesis、DecisionInput candidate、candidate/final 对比和 gate 结果。因此现阶段还不能说页面已经能看到完整真实的多 Agent 全链路交互。

这两个问题都属于主流程问题，不属于边缘优化。后续验收口径必须调整为：

- 没有把 Agent Swarm 变成可审计的主候选链路，不能判定 Agent 系统重构完成。
- 没有前端 Agent 可视化审计页面，不能判定全链路可观测完成。
- 没有启动前后端并做运行态页面自测，不能判定前端能力完成。
- 页面必须能让评审直接看到一次运行中 LeadAgent 如何拆任务、每个 Worker Agent 做了什么、用了哪些 Skill/tool、输出了哪些贡献、Harness 如何校验、Lead 如何综合、DecisionInput 如何生成、最终 gate 如何处理。

问题产生原因记录：

- 前期执行重点偏向后端结构拆分、包归属、兼容 wrapper、候选 artifact 和测试收敛，导致“真实主链是否已经切换”和“页面是否能直观看到 Agent 编排价值”没有被作为第一验收标准。
- 文档和测试覆盖了大量 sidecar/audit 产物，但验收没有强制要求从产品页面验证完整 Agent 交互，因此出现了后端有材料、前端不可见的断层。
- 只做代码级核对不能替代运行态自测。后续涉及前端页面或可观测性，必须启动服务并用真实 trace 验证页面展示。

新增主流程问题记录：

- 问题三：必须把“一次用户 query/manual run 如何触发完整后端流程”写成可读、可验证、可页面追踪的数据流。入口、上下文、每个 workflow step、Agent 编排、Skill/tool 调用、gate、final、persistence、notification 边界都必须有明确 owner 和 trace ref，不能让主流程变成只能从代码里猜的黑盒。
- 问题四：前端已经存在运行详情页，但验收不能停留在“有页面”。页面必须是可视化审计视图，而不是只显示运行摘要、span 列表和原始 JSON。评审应该能在页面上直接看懂 Agent 分工、执行顺序、并发关系、贡献内容、冲突处理、质量门禁和最终输入选择。
- 问题五：后续任何涉及 Agent 系统的重构，第一验收标准必须是主流程清晰和可控，而不是文件数量、测试数量或 sidecar artifact 数量。没有主流程图、运行态 trace、前端审计视图和可回放输入，不能认为 Agent 系统已经完成。
- 问题六：前端能力必须做运行态自测。至少要启动 API 和 frontend，用一次真实或 fixture run 生成 trace，再打开 `/runs/{trace_id}` 检查页面是否展示完整 Agent Swarm 审计链路。只跑 TypeScript typecheck 或 Next build 不能证明页面满足产品验收。
- 问题七：当前 `query_text` 虽然会进入 `DecisionRequest` 和 run context，但默认主链并没有把 query 显式转成任务计划、Agent 分配和 Skill/tool 调用策略。后续必须让 query 驱动的 intent、任务拆解、worker selection、tool budget、事实需求和最终输入选择成为可见流程，而不是只把 query 当作附属文本保存在上下文里。

下一步主线整改方向：

- 先画清并固定“一次 query/manual run 的真实数据流”，包括 API 入参、DecisionRequest、DecisionRunContext、workflow step、LeadPlan、Worker Agent、Skill/tool、DecisionInput、FinalDecisionAgent、gate、persistence、frontend detail view。
- 先补齐后端可观测投影，让 API 能返回完整但脱敏的 Agent Swarm 审计视图，而不是只返回 plan_run 摘要。
- 先补齐前端 Agent 可视化审计页面，让评审不读代码也能看懂一次运行中的 Agent 编排和质量门禁。
- 再评估是否把 Agent Swarm 从 pre-final sidecar 推进为主候选链路。这个切换必须以 query-driven planning、DecisionInput-only candidate final、release gate 和运行态可视化验收为前置条件。
- 暂停新增无主链接入的新模块、新 wrapper 和大范围测试堆叠。后续每个改动都必须能回答：它是否让 query 主流程更清晰、更可控、更可视化。

## 2.1 对抗审查结论

本节记录 4 个只读审查 Agent 的一致结论，作为后续计划和实现的硬输入。

后端主流程审查结论：

- 当前真实主链仍是 API -> `DecisionRequest` -> `DecisionRunContext` -> legacy workflow -> legacy prompt final -> parser -> production/risk gate -> persistence。
- `pre_final_orchestration` 和 shadow swarm 是 sidecar/audit，但其 candidate audit 会进入 `production_control_gate`，因此不是“完全无影响”，而是“audit-only input, production-blocking gate”。
- `query_text` 当前只被归一化、携带、追踪和持久化，并没有驱动 intent、任务拆解、Agent 分配、Skill/tool 策略或最终决策输入。
- `RunExecutor` 通过最近 traces 反查 `trace_id` 存在并发错配风险，runner/adapter 应显式返回本次 trace_id。
- `controlled_shadow` 仍不可作为完整生产替代链路，trace/persistence 绑定不完整。
- `alert_channel` 当前是伪控制字段，API 接收但通知实际由 config/notifier 决定。

Agent/Skill 编排审查结论：

- 当前命名角色包括 7 个 shadow worker、4 个 legacy reviewer 名称、1 个 `FinalDecisionAgent` 名称和 1 个 `LeadAgent`；真正业务 worker 是 7 个。
- 7 个 Worker 会执行，但都是 `decision_effect=none` 的 shadow audit，不直接替代 FinalDecisionAgent。
- `FinalDecisionAgent` 主链实际由 `DecisionEngine` 直接执行，不经过 AgentRunner/Harness。
- `skills/facade.py` 的 5 个 Skill 是合同边界，不是真实默认主链中的实时执行器；`RealtimeSearchSkill` 只读取 input view 里的 search results。
- Agent 之间有共享输入，但没有共享黑板、消息总线、多轮辩论、互评循环；当前是并发 copy input + 后处理 synthesis。
- `LeadAgent` 只是 shadow/audit 任务规划和综合者，不运行 worker、不调工具、不做最终交易决策。

前端和可观测性审查结论：

- 当前详情页更像 Trace 详情/复盘页，不是完整 Agent 可视化审计页。
- 页面能看到 trace 摘要、span、LLM 交互、parsed plan、verdict、badcase，但看不到 `shadow_swarm_audit`、`worker_results`、`lead_synthesis`、`DecisionInput`、`gate_candidate`、`final_decision_switch_readiness`、`audit_only` 等核心审计链路。
- 后端已经把完整 payload 存入 `plan_runs.payload_json`，但 `GET /api/runs/{trace_id}` 只通过 row projection 返回摘要；`include_payloads=true` 只影响 LLM request/response。
- 前端 schema 也没有声明 Agent audit 字段，后端即使临时透传，当前 schema 也可能剥掉未声明字段。
- 源码 UTF-8 正常，PowerShell 默认读取导致的中文乱码不是文件真实乱码；真正问题是展示依赖 JSON `<pre>`，没有 Agent 审计专用视图。

计划约束审查结论：

- `32` 已经写出主问题，但唯一执行入口 `31` 之前全 `[x]`，会误导后续认为主线已经完成。
- 必须在 `31` 新增未完成 Checkpoint 9，作为后续唯一执行入口。
- `29`、`30` 有过期事实：仍出现 4 worker 或 `agent_swarm/local_workers/` canonical owner 的表述，必须修正。
- 新增 sidecar/artifact 前必须同时明确 producer、persistence、API projection、frontend view、runtime smoke assertion，否则仍会继续堆旁路而不解决主流程。

## 3. 重构目标

目标不是把项目包装成“看起来有很多 Agent”，而是让系统能稳定做到：

- 能把复杂加密货币趋势判断拆成明确任务。
- 每个 Worker Agent 只负责一个可审计维度。
- 每个 Skill 只提供受控工具结果，不直接给最终交易动作。
- LeadAgent 能分配任务、汇总冲突、输出结构化综合意见。
- FinalDecisionAgent 只能基于通过 gate 的 `DecisionInput` 生成候选结论。
- 生产切换必须由 release gate、人工 config-change review 和回滚计划共同决定。
- 所有候选输出都可回放、可追踪、可审查、可对比 legacy。

目标主链：

```text
RunExecutor
  -> DecisionRunContext
  -> Workflow step runner
  -> realtime facts / evidence packets
  -> LeadAgent plans SubTasks
  -> Worker Agents run in parallel
  -> Skill facade returns controlled tool results
  -> Workers produce AgentContribution
  -> Harness validates constraints
  -> LeadAgent synthesizes conflicts and missing facts
  -> DecisionInput candidate
  -> DecisionInput input gates
  -> candidate FinalDecisionAgent
  -> parser / semantic / risk / release / switch readiness gates
  -> sidecar comparison with legacy
  -> manual release review
  -> controlled production switch only after approval
```

Gate 分层固定为：

- FinalDecisionAgent 前置输入 gate：`DecisionInput` schema/readback、required worker refs、execution fact source、hard block propagation、side-effect guard。
- FinalDecisionAgent 后置输出 gate：parser、semantic、risk、release gate、production switch readiness。

## 4. 不变的硬约束

- 不自动打开 `decision.final_input_mode=decision_input`。
- Checkpoint 7 之前，候选链路只能写 sidecar/audit 产物。
- shadow/candidate/eval/replay 不能写生产 journal、不能发生产通知、不能触发 live order。
- 业务 Agent 只能放在 `src/crypto_manual_alert/market_agents/`。
- `agent_swarm/` 只能承载 runtime、pool、registry、compat exports、LLM/tool worker runtime。
- Skill 不能返回 `AgentContribution`，不能生成 final decision，不能写副作用。
- LeadAgent 不能绕过 FactsGate、RiskGate 或生产控制 gate。
- FinalDecisionAgent 后续只能消费通过前置输入 gate 的 `DecisionInput`。
- 项目根目录不新增业务 `.py` 文件。
- `tests/` 根层不新增散落测试，按领域进入 `tests/<domain>/`。
- 同一个问题连续失败 3 次必须暂停，记录失败现象、已尝试方案、证据和需要用户决策的点。

## 6. Agent 与 Skill 目标拆分

### 6.1 Worker Agent

当前 required worker 保持 7 个：

- `LiveFactAgent`：只消费实时行情、snapshot、facts gate，判断关键事实是否可用。
- `DerivativesAgent`：只消费衍生品事实，分析资金费率、持仓、爆仓、基差等。
- `MacroEventAgent`：分析宏观事件状态、预期差、实际值、市场反应和滞后确认。
- `RootCauseAgent`：从事件出发递归展开影响因素，形成可审计因果链。
- `MarketSentimentAgent`：区分客观事实、拥挤交易、priced-in 和短期 reflexivity。
- `DataQualityAgent`：评估源质量、时效、冲突和执行事实覆盖。
- `ExecutionRiskAgent`：输出 hard block、人工复核提醒、可行动作类别收缩。

后续是否新增 `FlowLiquidityAgent` 需要在 Checkpoint 4G 单独决策。不能在未定义验收标准时顺手新增。

### 6.2 Skill

Skill 是工具，不是决策者。后续 Skill 设计保持以下类型：

- `RealtimeSearchSkill`：实时 web searched 信息检索，只能产生 search-derived evidence。
- `RootCauseSearchSkill`：按事件影响因素递归检索，必须限制深度、超时、来源和分支数量。
- `MarketSentimentSkill`：检索情绪、拥挤、反身性和 priced-in 线索，不能直接给交易动作。
- `MacroEventSkill`：结构化宏观事件状态、实际值、预期值、预期差和滞后确认。
- `LiquidityOrderBookSkill`：只接受交易所原生事实源，才可能满足 execution fact。

Skill 输出进入 Worker 前必须通过结构化边界。Worker 可以消费 tool result 并生成 `AgentContribution`，但禁止把原始 `SkillToolResult` 对象或 public dict 嵌入 `AgentContribution` 或 final decision；只能转成 normalized evidence refs 或 facts 后再由 Worker 生成 contribution。

## 7. 记忆与 Harness 的处理方式

### 7.1 记忆机制

记忆机制有价值，但不能优先于事实链路。

短期记忆建议保留为会话上下文能力，最多用于：

- 用户多轮追问的意图补全。
- 本轮用户偏好、风险偏好和关注标的。
- 最近几轮已经解释过的上下文摘要。

长期记忆建议后置到质量门禁稳定之后，最多用于：

- 相似历史案例检索。
- badcase lesson 检索。
- 用户偏好和复盘经验检索。

记忆不能用于：

- 替代实时事实。
- 替代 execution fact。
- 绕过 FactsGate、RiskGate、DataQualityAgent 或 ExecutionRiskAgent。
- 把旧结论直接注入本轮 final decision。

### 7.2 Harness Engineering

Harness 是必要能力，但要保持轻量受控。

必须做：

- 通过显式 policy 定义 Agent 能力边界。
- 校验 forbidden fields、decision_effect、status、required worker、tool policy。
- 对 shadow/candidate/eval/replay 做 side-effect guard。
- 对 `AgentContribution` 和 `DecisionInput` 做 schema/readback 校验。

谨慎做：

- 自动修复只能用于候选 sidecar 或格式修正，不能自动修复生产交易含义。
- YAML policy 可以做配置来源，但核心资金安全规则必须有代码硬门禁和测试。

不做：

- 让 LLM 自己决定是否放开生产 gate。
- 让 harness 自动把失败 candidate 提升为生产结果。
- 只靠 prompt 约束 Agent 能力边界。

## 8. 分阶段计划

### 阶段二：建立候选 FinalDecisionAgent

目标：让候选 final 只消费通过前置输入 gate 的 `DecisionInput`，并与 legacy final 可对照回放。

## 9. 代码可读性与维护性清单

每个后续小闭环都必须检查：

- 新增文件职责一句话能说清。
- 新增文件放在业务归属目录，不放根目录。
- 文件名体现业务含义，不用临时、实验、版本号命名。
- 一个文件只承担一个主要职责。
- 修改大文件时，优先提取稳定边界，不做无关重排。
- 新增 public API 有对应测试。
- 兼容导出必须有生命周期说明。
- 旧入口被替代时必须有 adapter 或迁移说明。
- 不新增只有文档提到、代码没有接入的“空实现”。
- 不新增只有测试引用、真实链路没有引用的“假完成”。

## 10. 功能质量清单

每次涉及业务判断能力时必须检查：

- 是否区分实时事实、历史记忆、LLM 推理和人工偏好。
- 是否区分 search-derived evidence 与 exchange-native execution fact。
- 是否有 stale、missing、conflict 的处理。
- 是否传播 confidence cap 和 cap reason。
- 是否传播 hard block 和 manual review reminder。
- 是否有 counter thesis 或反方链。
- 是否能解释为什么某些动作被禁止。
- 是否能回放当时输入，不依赖当前实时环境重跑。
- 是否没有把候选结论伪装成生产结论。

## 11. 汇报和暂停规则

每完成一个小闭环必须汇报：

```text
当前阶段:
当前小闭环:
完成项:
修改文件:
失败测试或结构护栏是否先写:
新增文件归属:

生产默认链路是否仍为 legacy_baseline + legacy_prompt:
验证命令:
结果:
未完成项:
是否触发暂停条件:
下一步:
```

暂停条件：

- 同一个测试失败连续 3 次。
- 同一个导入边界或循环依赖连续 3 次。
- 同一个设计冲突连续 3 次。
- 为了通过测试需要删除关键测试、绕过结构护栏或扩大范围。
- 发现切换生产输入、生产通知、生产 journal、live order 边界被破坏。

暂停时必须写清：

- 失败现象。
- 已尝试方案。
- 当前证据。
- 可选方案。
- 需要用户决策的点。

## 12. 最终完成定义

本轮重构不能只以“测试通过”作为完成，必须同时满足：

- 生产默认仍可回到 legacy。
- 候选 Agent Swarm 主链能完整跑通。
- 7 个 required worker 的贡献进入 `DecisionInput`。
- Candidate final 只消费 `DecisionInput`。
- Release gate 能解释是否允许切换。
- 切换需要人工 review artifact。
- 回滚路径可执行。
- 根目录和根包没有新增业务 `.py`。
- 旧兼容层有生命周期说明。
- README 和 formal index 能说明当前架构。
- 全量测试通过或明确记录未通过原因和剩余风险。

# v1 需求边界

## 背景

本项目目标是构建一个个人使用的加密货币手动操作提醒系统。

系统根据固定的 `crypto-macro-decision` skill 和实时市场事实，生成一个可执行但需要用户手动确认的操作计划，并通过 Bark / 飞书提醒用户。

v1 不做自动交易。

## v1 目标

v1 要实现：

- 支持手动 query 触发一次评估。
- 支持定时任务按配置触发评估。
- 固定加载 `crypto-macro-decision` skill。
- 使用 skill 的 `SKILL.md`、必要 references 和 `scripts/okx_snapshot.py`。
- 每轮刷新实时市场事实。
- 构造最小 `EvidenceSnapshot`。
- 生成主根因链和反向根因链。
- 生成唯一 `DecisionPlan`。
- 严格 JSON 解析。
- 通过代码级 `RiskGate` 执行 hard block / soft downgrade / confidence cap。
- 写入 journal。
- 发送 Bark 通知，飞书作为可配置后续通道。

## v1 不做

v1 不做：

- 自动下单。
- 交易 API key 接入。
- 提现 API key 接入。
- 7-agent 并行 swarm。
- 任意 skill 自动发现。
- 任意脚本动态注册工具。
- 长期向量记忆。
- 从 journal 默认检索旧结论参与 live decision。
- 复杂回测平台。
- 多交易所执行适配。
- 仓位自动读取和自动计算真实下单数量。

## 用户入口

### 手动 query

示例：

```text
ETH 当前持有多单，重新评估 6 小时，现在怎么操作？
```

系统需要识别：

- symbol
- horizon
- 当前持仓方向
- 用户意图

如果关键字段缺失，最多问一个澄清问题。

### 定时 query

示例：

```yaml
scheduled_queries:
  - name: eth_6h
    symbol: ETH-USDT-SWAP
    horizon: 6h
    position_state: long
    interval_seconds: 1800
```

定时任务必须显式配置，不允许隐式推断仓位。

## 输出要求

系统最终输出一个 `DecisionPlan`。

`main_action` 必须是以下枚举之一：

- `open long`
- `open short`
- `hold long`
- `hold short`
- `close long`
- `close short`
- `flip long to short`
- `flip short to long`
- `trigger long`
- `trigger short`
- `no trade`

输出必须包含：

- instrument
- horizon
- reference price
- entry trigger
- stop price
- target 1
- target 2
- probability
- confidence cap reason
- why not opposite
- unavailable data
- manual execution required
- expiry

## 成功标准

一次成功运行必须满足：

- 已加载固定 skill。
- 已记录 skill hash。
- 已刷新市场事实。
- 已明确数据缺口。
- 已生成根因主链和反链。
- 已生成唯一计划。
- 已通过严格解析。
- 已通过或被风控明确拦截。
- 已写入 journal。
- 通知成功或通知失败被记录。

## 失败标准

以下情况必须 fail closed：

- skill 文件缺失。
- 核心事实缺失且无法兜底。
- LLM 超时或不可用。
- 输出不是严格 JSON。
- 输出缺少必需字段。
- 输出违反 action enum。
- 新开仓没有 stop。
- probability 超过代码计算的 confidence cap。
- journal 写入失败。

# 成熟观测与评测平台接入方案

> **当前定位（2026-07-09）**：本文是 post-v1 proposal，不进入当前 P0/P1 执行主线。当前交付执行、P0 证据闭环和 checklist 以 `docs/implementation/2026-07-09-current-delivery-checklist.md` 为准；Langfuse/DeepEval 等成熟平台接入在 prod-actionable、production-intent hosted runtime 和 real-outcome gate 关闭前暂停。

日期：2026-07-06

## 1. 结论

建议采用“成熟平台做通用 LLM/Agent 观测与评测，本项目只保留交易域控制面”的混合架构：

- Langfuse 作为主观测平台：承接 trace、span、LLM call、tool call、token/cost、scores、annotation、dataset、experiment。
- DeepEval 作为离线/CI eval 框架：承接 agentic metrics、tool correctness、plan adherence、JSON/schema/safety 等回归测试。
- 本项目自研 UI 降级为交易域 Cockpit：只展示生产能不能信、为什么不能执行、缺哪些执行事实、legacy/candidate/gate 差异、financial quality 状态，并跳转到 Langfuse trace / eval run。
- facts gate 与 financial quality gate 保持本项目内生，不交给 Langfuse/DeepEval 决定是否允许交易。

这不是引入更多复杂度，而是减少自研观测 UI 的长期维护成本。

## 2. 背景问题

当前 Run Detail 已经能投影 Agent Swarm 主链路，但可读性不足：

- 大量表格和 JSON 并列，首屏信息密度过高。
- `AgentAuditPanel` 既承担业务驾驶舱，又承担 raw trace browser。
- `facts_gate`、`financial_quality_gate`、`source_freshness`、`candidate_comparison` 混在一起，非开发者不容易判断“到底能不能信”。
- 自研 LLM/Agent observability 容易重复 Langfuse/Phoenix/LangSmith 这类成熟工具已有能力。
- 自研 eval 页面也容易重复 DeepEval/Ragas/OpenAI Evals 等框架已有能力。

当前 trace 的具体问题也说明了边界：

- `facts_gate` 缺 `mark/index/order_book`，这是执行事实缺失。
- `search_derived` 可以解释事件，但不能满足执行事实。
- `financial_quality_gate=not_configured` 是 outcome store / 历史样本闭环没配置，不是 web search 没接通。

## 3. 约束

必须保持：

- 默认 `workflow.execution_mode=legacy_baseline`。
- 默认 `decision.final_input_mode=legacy_prompt`。
- 默认 `shadow.worker_mode=local_audit`。
- 不默认启用 Langfuse、DeepEval、真实 web search、真实外部 LLM、真实交易 provider。
- 不自动交易，不下单、不撤单、不提现、不发交易通知。
- search-derived evidence 不能满足 `mark/index/order_book` execution facts。
- FinalDecisionAgent 不得调用 tool。
- Worker 输出不得包含最终交易动作字段。

## 4. 成熟平台职责划分

### 4.1 Langfuse

Langfuse 用于通用 LLM/Agent observability：

- trace / session / observation；
- nested spans；
- LLM prompt/response；
- token、latency、cost；
- tool execution；
- scores；
- annotation；
- datasets；
- experiments；
- dashboards。

官方文档说明 Langfuse 的 tracing 会记录 prompt、model response、token usage、latency、tool/retrieval steps，并且支持 evaluation scores、datasets、experiments；也支持 OpenTelemetry ingestion。

参考：

- https://langfuse.com/docs/observability/overview
- https://langfuse.com/docs/evaluation/overview
- https://langfuse.com/integrations/native/opentelemetry

Langfuse 不负责：

- 判断 `mark/index/order_book` 是否可满足交易执行事实；
- 判断是否允许生产 final input；
- 计算金融结果 hit rate / Brier / PnL / MFE / MAE；
- 存储任何交易密钥或未脱敏 payload。

### 4.2 DeepEval

DeepEval 用于离线/CI eval：

- Task Completion；
- Tool Correctness；
- Argument Correctness；
- Step Efficiency；
- Plan Adherence；
- Plan Quality；
- RAG / generator / safety / custom metrics。

官方文档说明 DeepEval 的 agentic metrics 覆盖 Task Completion、Argument Correctness、Tool Correctness、Step Efficiency、Plan Adherence、Plan Quality，并支持 custom metrics。

参考：

- https://deepeval.com/docs/metrics-introduction
- https://docs.confident-ai.com/docs/getting-started

DeepEval 不负责：

- 替代 release gate；
- 替代 financial quality outcome collector；
- 替代交易执行事实门禁；
- 线上请求同步阻断。

### 4.3 本项目本地控制面

本项目继续负责交易域强语义：

- production gate；
- facts gate；
- source freshness；
- execution fact refs；
- candidate vs legacy diff；
- final input selection；
- manual execution required；
- financial quality gate；
- release review evidence bundle；
- rollback checklist。

## 5. 推荐架构

```text
RunExecutor
  -> ObservabilityRecorder
      -> Local Journal Sink (default, current SQLite)
      -> Optional Langfuse Sink (explicit config)
      -> Optional OTLP Sink (future)

EvalRunner
  -> Local EvalStore / OutcomeStore
  -> Optional DeepEval Runner (offline/CI)
  -> Optional Langfuse Scores Exporter

Frontend
  -> Local Trading Cockpit
  -> Link out: Langfuse Trace URL
  -> Link out: DeepEval/Eval Run Artifact
```

核心原则：

- 本项目保留 canonical local ledger，不把 Langfuse 当唯一事实源。
- Langfuse 是 exporter / mirror，不是生产主存储。
- DeepEval 是 offline test runner，不进线上同步决策路径。
- 所有外部平台都显式启用。

## 6. 数据映射

### 6.1 Trace

本地字段：

- `traces.trace_id`
- `trace_spans.span_id`
- `llm_interactions.id`
- `ToolCallArtifact.tool_call_id`
- `agent_audit_view`

Langfuse 映射：

- trace name：`manual_run` / `production_candidate_swarm`
- trace id：复用本地 `trace_id`
- environment：`local` / `shadow` / `candidate` / `prod`
- session id：同一人工分析会话可选绑定
- metadata：
  - symbol
  - horizon
  - execution_mode
  - final_input_mode
  - candidate_status
  - production_final_input
  - facts_gate_status
  - financial_quality_status

### 6.2 Span / Observation

本地 `trace_spans` 映射为 Langfuse span：

- `market.fetch`
- `research.search`
- `lead.plan`
- `shadow_swarm.worker`
- `skill.execute`
- `candidate.final`
- `production_control_gate`
- `persistence`

本地 `llm_interactions` 映射为 Langfuse generation：

- model；
- endpoint；
- prompt/input summary；
- sanitized request；
- sanitized response；
- token usage；
- cost；
- duration；
- finish reason；
- retry count；
- error type。

ToolCallArtifact 映射为 Langfuse span：

- skill name；
- source type；
- source tier；
- freshness；
- fact refs；
- can_satisfy_execution_fact；
- error type/hash。

### 6.3 Scores

本地 gate 映射为 Langfuse score：

- `facts_gate.passed`
- `facts_gate.confidence_cap`
- `production_control.allowed`
- `candidate_final.allowed`
- `financial_quality.status`
- `symbol_consistency`
- `side_effect_proof`
- `manual_execution_required`

注意：Langfuse score 是可视化和筛选信号，不是生产放行来源。

### 6.4 DeepEval TestCase

从本地 trace 构建 DeepEval cases：

- input：冻结后的 query、symbol、horizon、market snapshot summary；
- actual output：candidate final / worker summaries / selected final input；
- expected output：badcase 人工标签或离线 ground truth；
- tools called：ToolCallArtifact 列表；
- retrieval context：search_derived refs；
- metadata：facts gate、source freshness、release gate。

## 7. Run Detail UI 重构方向

当前页面应从“trace dump”改为“交易域 Cockpit”：

### 7.1 第一屏

只显示：

- 能否用于生产 final input：yes/no；
- 当前模式：legacy / controlled_shadow / production_candidate_swarm；
- 阻断原因：top 3；
- 缺失执行事实：mark/index/order_book；
- Candidate vs Legacy 结论；
- Tool health；
- Financial quality 状态；
- Langfuse trace link；
- Eval/DeepEval run link。

### 7.2 第二屏

显示业务矩阵：

- Worker Matrix；
- Skill Tool Calls；
- Source Freshness；
- Candidate Comparison；
- Facts Gate；
- Release Gate。

### 7.3 第三屏

只作为展开辅助：

- raw JSON；
- sanitized LLM payload；
- span summaries；
- replay refs；
- badcases。

## 8. Provider 与数据质量策略

web search 更适合：

- 当前事件；
- 新闻确认；
- ETF / 宏观事件；
- 交易所公告；
- 链上/舆情解释；
- root cause 辅助。

web search 不适合：

- mark price；
- index price；
- order book；
- funding exact value；
- open interest exact value；
- execution-ready liquidation level。

生产策略：

- exchange-native provider 优先；
- 可以配置多个行情源做冗余；
- web search 只能生成 `search_derived`；
- exchange API 不通时，允许输出“分析-only/no opening action”，不允许 opening/trigger/flip；
- facts gate 缺执行事实时仍 hard fail。

## 9. Financial Quality 策略

financial quality 不是实时 provider 问题。

它需要：

- outcome collector；
- trace/run close；
- horizon 到期后回收价格；
- hit rate；
- Brier score；
- PnL/R multiple；
- MFE；
- MAE；
- regime slice；
- legacy/candidate/no-trade baseline。

Langfuse/DeepEval 可以展示或辅助评测，但不能替代 outcome collector。

## 10. 分阶段计划

### Phase A：方案冻结

产物：

- 本方案；
- 多角色对抗审查；
- 35 文档更新；
- 不改代码。

验收：

- 用户确认采用混合架构；
- 明确 Langfuse 是否 self-host；
- 明确 DeepEval 是否作为 CI 必选。

### Phase B：Run Detail 收缩

目标：

把当前 Run Detail 从全量 dump 收缩为 Cockpit。

主要文件：

- `frontend/src/app/runs/[traceId]/page.tsx`
- `frontend/src/app/runs/[traceId]/agent-audit-panel.tsx`
- `frontend/src/app/runs/[traceId]/tool-call-graph.tsx`
- `frontend/src/app/styles.css`
- `tests/structure/test_frontend_route_boundaries.py`

验收：

- 首屏只展示 production readiness；
- JSON 默认折叠；
- 桌面/移动端无横向溢出；
- Playwright screenshot 通过。

### Phase C：Observability Sink 抽象

目标：

让 `ObservabilityRecorder` 支持 fanout sink。

新增/修改：

- `src/crypto_manual_alert/telemetry/sinks.py`
- `src/crypto_manual_alert/telemetry/local_sink.py`
- `src/crypto_manual_alert/telemetry/langfuse_sink.py`
- `src/crypto_manual_alert/config/models.py`
- `src/crypto_manual_alert/config/loader.py`
- `tests/telemetry/`

设计：

- default：local only；
- explicit config：`observability.exporters=["langfuse"]`；
- key 来自 env；
- exporter failure 不影响主流程；
- exporter 只发送脱敏 payload/hash/summary。

### Phase D：Langfuse Exporter

目标：

把本地 trace/span/LLM/tool/gate 镜像到 Langfuse。

验收：

- 生成一次 manual run；
- Langfuse 能看到同 trace id；
- span 层级可读；
- tool call 可筛选；
- gate score 可筛选；
- 本地 SQLite 仍完整。

### Phase E：DeepEval Offline Runner

目标：

把 eval cases 接入 DeepEval。

新增：

- `src/crypto_manual_alert/eval/deepeval_runner.py`
- `src/crypto_manual_alert/eval/deepeval_cases.py`
- `tests/eval/test_deepeval_runner.py`

首批 metrics：

- Tool Correctness；
- Plan Adherence；
- Task Completion；
- JSON/schema correctness；
- side-effect safety；
- custom trading-domain rubric。

验收：

- 无外部 key 时 skip/fail closed；
- fixture judge 可跑 CI；
- real judge 显式配置才跑；
- eval result 回写本地 EvalStore；
- 可选写入 Langfuse score。

### Phase F：Financial Outcome 闭环

目标：

完成金融质量 gate 的真实样本来源。

重点：

- outcome collector；
- horizon maturity；
- historical price close；
- metrics；
- baseline compare；
- release gate advisory evidence。

验收：

- `financial_quality_gate` 不再长期 `not_configured`；
- 不足样本显示为 `not_enough_samples`；
- 样本充足后显示 pass/fail；
- 不影响线上安全 gate。

### Phase G：Release Review 汇总

目标：

把 Langfuse/DeepEval/FinancialQuality/本地 gate 合并为 release review evidence bundle。

验收：

- release report 含 trace links；
- 含 DeepEval run id；
- 含 financial metrics；
- 含 rollback plan；
- 仍需人工确认才允许切 final input。

## 11. 配置草案

```yaml
observability:
  exporters:
    - local
  langfuse:
    enabled: false
    host: ""
    public_key_env: LANGFUSE_PUBLIC_KEY
    secret_key_env: LANGFUSE_SECRET_KEY
    send_payloads: false
    send_scores: true
    timeout_seconds: 5

eval:
  deepeval:
    enabled: false
    mode: fixture
    judge_model: ""
    api_key_env: OPENAI_API_KEY
    write_langfuse_scores: false
```

## 12. 安全与隐私

- 任何 exporter 默认关闭。
- 所有 key 从 env 读取，不进 config safe dict。
- 不发送 raw secret、交易 key、设备 key。
- 默认只发送 hash、summary、refs。
- `send_payloads=true` 需要单独配置和测试。
- exporter 异常只能写观测 error，不阻断主流程。
- release gate 不读外部平台作为唯一事实源。

## 13. 对抗审查

### 13.1 架构审查 Agent

结论：推荐混合架构。

阻断意见：

- 如果 Langfuse 成为唯一 trace store，会削弱本地 replay 和安全审计。
- 如果直接把所有本地字段发到 Langfuse，会造成隐私和成本问题。

整改：

- 保留 local canonical ledger。
- Langfuse 只做 mirror/exporter。
- exporter 必须 fail-open，不影响主流程。

### 13.2 交易安全审查 Agent

结论：Langfuse/DeepEval 不能替代交易 gate。

阻断意见：

- web search 不能满足 mark/index/order_book。
- DeepEval 不能决定是否允许交易。
- Langfuse score 不能作为生产放行来源。

整改：

- facts gate 仍由本地代码计算。
- financial quality 仍由 outcome collector 计算。
- release review 必须人工确认。

### 13.3 Eval 科学审查 Agent

结论：DeepEval 适合 agent/tool 行为评测，不适合直接证明金融收益。

阻断意见：

- LLM-as-judge 不能替代真实 outcome。
- 没有 ground truth 的线上 eval 只能做参考。

整改：

- DeepEval 用于 tool correctness、plan adherence、task completion。
- 金融质量单独用 outcome metrics。
- 每个 metric 上限 5 个以内，避免评测噪音。

### 13.4 前端产品审查 Agent

结论：当前 Run Detail 需要大幅收缩。

阻断意见：

- 页面现在像开发者 dump，不像运营驾驶舱。
- 表格太多，首屏不回答“能不能信”。

整改：

- 首屏只保留 decision readiness。
- 原始数据折叠。
- 链接跳转 Langfuse trace。

### 13.5 运维成本审查 Agent

结论：引入平台必须分级。

阻断意见：

- self-host Langfuse 有部署、升级、备份成本。
- DeepEval real judge 会引入额外 LLM 成本。

整改：

- 先做 exporter abstraction。
- local-only 默认。
- Langfuse/DeepEval 显式开关。
- CI 先用 fixture/deterministic metrics。

### 13.6 代码维护审查 Agent

结论：不能把 Langfuse SDK 调用散落各处。

阻断意见：

- 如果每个 worker/tool 直接调 Langfuse，会形成强耦合。

整改：

- 统一通过 `ObservabilitySink`。
- 统一 schema mapping。
- 单测覆盖 redaction、failure isolation、config disabled。

## 14. 最终推荐

采用 Phase B -> C -> D -> E -> F -> G。

先改 UI，快速降低混乱；再接 Langfuse exporter；再接 DeepEval；最后补金融 outcome。

不建议：

- 直接删除本地 trace/eval。
- 直接让 Langfuse/DeepEval 参与生产放行。
- 继续扩大自研 observability UI。
- 用 web search 代替 execution facts。

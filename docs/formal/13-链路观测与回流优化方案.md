# 链路观测与回流优化方案

## 1. 结论

当前不建议立刻实现完整的观测平台，也不建议接入复杂的外部 LLMOps 系统。

但必须从现在开始把链路观测作为架构边界设计好，并在关键位置预留 `trace_id`、`span`、反馈归因和离线评估接口。原因是：

- 当前基础交易提醒能力还在演进，过早建设完整平台会拖慢主线。
- 但 agent 链路已经包含 Leader planner、并发 researcher、Leader reviewer、final decision、RiskGate、Bark，黑盒程度已经足够高。
- 如果现在不设计观测结构，后续很难判断错误来自 planner、搜索、证据合成、leader 审查、最终模型、parser、风控还是通知。
- 最小观测层成本不高，可以先记录结构化事件，不影响主流程。

推荐路线：

```text
现在：设计 Trace Ledger 接口 + 文档化 span taxonomy + 关键字段预留
下一步：实现本地 SQLite trace 表和 CLI 查询
后续：增加用户反馈归因、离线 replay/eval
成熟后：可选导出到 OpenTelemetry / Langfuse / Phoenix
```

## 2. 参考原则

本方案参考以下公开架构方向：

- OpenAI Agents SDK tracing：一次 agent 运行由 trace 表示，内部包含 LLM generation、tool call、handoff、guardrail 和 custom span。
- OpenTelemetry GenAI semantic conventions：为 GenAI 请求、响应、模型、token、tool、agent、conversation 等字段提供通用语义。
- Langfuse：强调 trace、prompt、response、tool call、metadata、latency、cost、eval、dataset 的闭环。
- Phoenix：强调 trace 能看到 model call、retrieval、tool use、custom logic，并支持 datasets / experiments / evaluations。

本项目不直接照搬任何平台，但数据模型应尽量接近这些通用概念，避免未来迁移成本过高。

## 3. 当前系统缺口

当前已有：

- `plan_runs.payload_json` 记录 `snapshot`、`research`、`raw_decision`、`parsed_plan`、`verdict`。
- `research` 中有 `plan`、`results`、`leader_summary`。
- `notifications` 记录 Bark 发送结果。
- `manual_outcomes` 可以记录人工执行结果。

主要缺口：

- 没有 `trace_id` 贯穿一整轮任务。
- 没有 `span_id` / `parent_span_id` 描述阶段层级。
- 没有每个阶段的开始时间、结束时间、耗时、状态、错误类型、fallback 原因。
- 没有 LLM 请求的 token、模型、prompt 版本、输出摘要、输出 hash。
- 没有 tool/search 的单独调用记录和并发关系。
- 没有统一 failure taxonomy，无法长期统计错因。
- 没有把生产样本沉淀成 eval dataset。
- 没有 replay 机制，无法比较 skill/prompt/risk 规则改动前后的效果。

因此当前 journal 更像“最终大 JSON 留档”，不是生产级“链路观测系统”。

## 4. 设计目标

### 4.1 必须做到

- 每次 `run_once` 生成一个 `trace_id`。
- 每个关键阶段生成一个 span。
- span 记录状态、耗时、输入摘要、输出摘要、错误和 fallback。
- plan run、notification、manual outcome 都能关联到 `trace_id`。
- 支持用户或开发者对 trace 打反馈标签。
- 支持把高价值 trace 提取为 eval case。
- 默认脱敏，不把 API key、账户 key、完整敏感 prompt 写入日志。

### 4.2 现阶段不做

- 不接完整外部观测平台作为强依赖。
- 不做复杂 Web UI。
- 不做全量 prompt/completion 永久存储默认开启。
- 不做自动训练或自动改 prompt。
- 不让历史 trace 默认回灌 live decision。
- 不做交易绩效自动归因替代人工复盘。

### 4.3 分析过程记录边界

用户需要能回溯“系统为什么给出这个结论”，但系统不应保存或依赖模型隐藏思维链。后续回归与优化记录的是可审计分析摘要，而不是原始 chain-of-thought。

必须记录：

- `reasoning_summary`：模型或 reviewer 给出的简体中文推理摘要。
- `evidence_to_claims`：关键证据如何支持或削弱多头、空头、no trade、trigger 等结论。
- `decision_ladder`：从事实门、研究补充、对抗审查、最终决策到风控阻断的阶段结论。
- `risk_rule_hits`：confidence cap、缺核心行情、数据陈旧、parser 失败、通知失败等规则命中。
- `opposing_thesis`：最强反向根因链，以及为什么没有成为主操作。
- `data_gaps`：缺失、超时、陈旧、来源等级不足或互相冲突的数据。
- `prompt_version`、`skill_hash`、`model_name`、`config_hash`：用于判断同一轮回放是否具备可比性。

禁止默认记录：

- API key、Bark key、账户 key。
- 完整敏感 prompt 或完整用户隐私原文。
- 模型隐藏思维链或要求模型暴露“逐步内心推理”的文本。
- 历史 trace 默认回灌到 live decision，避免旧结论污染新行情。

实现要求：

- 对外通知、journal 摘要、reviewer summary 和风控原因默认使用简体中文。
- JSON key、枚举值、symbol、URL、source label、错误类型保持协议格式，不强行翻译。
- 如果某阶段无法产出摘要，要记录 `summary_unavailable_reason`，不能伪造分析过程。

## 5. Trace 模型

### 5.1 Trace

一条 trace 表示一次完整业务运行。

```text
trace_id
created_at
run_type: manual | scheduled | replay | eval
symbol
horizon
position_state
user_query_hash
user_query_summary
status: ok | blocked | failed
final_plan_id
final_action
final_allowed
total_duration_ms
skill_name
skill_hash
prompt_version
risk_rule_version
config_hash
metadata_json
```

说明：

- `user_query_summary` 只保存摘要，避免保存用户隐私。
- `config_hash` 用于判断同一批 eval 是否运行在相同配置下。
- `prompt_version` 和 `risk_rule_version` 后续应由代码常量维护。

### 5.2 Span

一条 span 表示 trace 内的一个阶段。

```text
span_id
trace_id
parent_span_id
span_name
span_type
started_at
ended_at
duration_ms
status: ok | error | fallback | skipped | blocked
input_summary_json
output_summary_json
input_hash
output_hash
model_provider
model_name
request_timeout_seconds
token_input
token_output
cost_estimate
error_type
error_message
fallback_reason
metadata_json
```

`span_type` 建议枚举：

```text
request.parse
skill.load
market.fetch
research.plan
research.search
evidence.synthesize
leader.review
decision.llm
parser.strict_json
risk.check
notification.send
feedback.record
eval.replay
```

### 5.3 Event

event 用于记录 span 内的重要点，不一定形成完整阶段。

```text
event_id
trace_id
span_id
created_at
event_name
level: info | warning | error
message
data_json
```

示例：

- `fallback.static_planner_used`
- `web_search.zero_tool_usage`
- `risk.confidence_cap_applied`
- `parser.invalid_json`
- `notification.bark_200`

### 5.4 Artifact

artifact 用于保存可选的大对象。默认不保存完整 prompt/completion，只保存 hash 和摘要。

```text
artifact_id
trace_id
span_id
artifact_type: prompt | completion | search_result | snapshot | plan | verdict
content_hash
content_summary
content_json
redaction_status
created_at
```

配置建议：

```yaml
observability:
  capture_full_content: false
  max_artifact_chars: 12000
  redact_secrets: true
```

## 6. 当前业务链路的 span 设计

```text
trace: run_once ETH-USDT-SWAP
  span request.parse
  span skill.load
  span market.fetch
  span research.plan
  span research.search: query_1
  span research.search: query_2
  span evidence.synthesize
  span leader.review
  span decision.llm
  span parser.strict_json
  span risk.check
  span notification.send
```

并发 researcher 的多个 `research.search` span 使用同一个 parent span：

```text
research.plan
  -> research.search(query=A)
  -> research.search(query=B)
  -> research.search(query=C)
```

这样后续可以清楚看到：

- 哪个 query 慢。
- 哪个 query 没结果。
- 哪个 query 触发 fallback。
- Leader reviewer 是否真的基于所有结果。

### 7.3 反馈记录

```text

```

## 8. 回流优化闭环

生产系统的目标不是“记录完就结束”，而是让记录进入优化循环。

```text
trace 收集
  -> 自动归因
  -> 用户/开发者反馈
  -> 高价值样本入 eval_cases
  -> 离线 replay
  -> 对比 prompt / skill / risk rule 版本
  -> 通过 eval 后再进入生产配置
```

### 8.1 Eval Case

```text
case_id
source_trace_id
dataset_name: golden_cases | failure_cases | market_outcome_cases
input_snapshot_hash
input_summary_json
expected_behavior
failure_category
created_at
```

### 8.2 Eval Run

```text
eval_run_id
dataset_name
candidate_version
baseline_version
started_at
ended_at
metrics_json
result_summary
```

核心指标：

```text
schema_valid_rate
manual_only_compliance
leader_review_completeness
source_url_presence
evidence_grounding_score
confidence_cap_compliance
risk_gate_false_allow
risk_gate_false_block
latency_by_stage
cost_by_trace
user_accept_rate
post_horizon_direction_accuracy
```

## 9. 是否现在实现

### 9.1 不建议现在实现完整版本

原因：

- 当前核心业务仍在调整，完整 trace/eval 平台会带来额外复杂度。
- 交易提醒链路还没稳定到需要大规模统计分析。
- 现在过早做 UI、dashboard、外部平台接入，收益不高。

### 9.2 建议现在实现最小可插拔版本

最小版本只做三件事：

1. 生成 `trace_id` 并写入 `plan_runs.payload_json`。
2. 提供 `TraceRecorder` 接口，但默认实现只写 SQLite。
3. 给关键阶段包一层 span context manager。

最小 API：

```python
trace = recorder.start_trace(run_type="manual", symbol=symbol)
with recorder.span(trace.id, "market.fetch", "market.fetch"):
    snapshot = market_provider.fetch_snapshot(symbol)
```

这不会明显拖慢功能开发，但后续可以平滑扩展。

## 10. 后续好不好改

如果按本方案做，后续好改。

关键原因：

- `TraceRecorder` 是接口，SQLite、OTLP、Langfuse、Phoenix 都可以作为不同 sink。
- 业务代码只调用 `start_trace`、`span`、`record_event`，不关心底层平台。
- trace schema 接近 OpenTelemetry / LLMOps 通用概念，未来迁移成本低。
- artifact 默认 hash + 摘要，后续可以按配置开启完整内容，不需要改业务链路。
- feedback / eval 是独立表，不会污染 live decision。

如果不按本方案预留，后续会难改：

- 只能解析大 JSON，无法稳定统计阶段耗时和错误。
- 无法知道多 agent 中哪个阶段出错。
- 旧数据无法回放。
- prompt/skill/risk rule 改动没有基准评估。

## 11. 分阶段落地计划

### 文档和接口设计

目标：

- 确认 trace/span/feedback/eval 数据模型。
- 确认哪些内容默认脱敏。
- 确认不接外部平台。

产出：

- 本文档。
- 后续实施计划。

### 本地 Trace Ledger

目标：

- 增加 SQLite 表：`traces`、`trace_spans`、`trace_events`、`trace_feedback`。
- `run_once` 生成 `trace_id`。
- market、research、leader、decision、parser、risk、notification 关键阶段记录 span。

验收：

- 能用 CLI 查看一条 run 的阶段耗时和错误。
- 失败时能定位具体阶段。

### 反馈归因

目标：

- 增加 `feedback` CLI。
- 支持手动打标签。
- 自动归因常见错误。

验收：

- 能统计最近 N 次失败中各类错误占比。
- 能按 `planner_error`、`grounding_error`、`risk_rule_block` 查询 trace。

### Eval Dataset 和 Replay

目标：

- 支持把 trace 提取为 eval case。
- 支持 fixture / frozen snapshot replay。
- 支持对比 prompt/skill/risk rule 版本。

验收：

- 改 prompt 前能跑 baseline。
- 改 prompt 后能对比 schema、grounding、risk compliance。

### 外部平台导出

目标：

- 可选导出 OpenTelemetry OTLP。
- 可选接 Langfuse 或 Phoenix。

验收：

- 本地 trace 数据不丢。
- 外部平台只是 sink，不成为系统强依赖。

## 12. 对当前项目的建议

当前最合理的下一步不是继续扩大 agent 数量，而是补最小 Trace Ledger。

优先级：

1. 修正源码中可能存在的中文注释编码显示问题，避免后续维护误判。
2. 加 `TraceRecorder` 接口和 SQLite 表。
3. 给 `PlanRunner.run_once` 加 trace。
4. 给 research planner / search / leader / final decision 加 span。
5. 增加 `trace-show` CLI。

不建议现在做：

- Web dashboard。
- 外部平台接入。
- 自动 LLM-as-judge 大规模评估。
- 复杂多用户权限。

先做到“每次运行都能回放链路、定位错因”，再谈“持续优化”。

## 13. 开源工具集成策略

### 13.1 总体判断

现成开源工具可以集成，但不建议让它们接管本项目的核心编排。

本项目的业务链路很强约束：

- 必须固定使用 `crypto-macro-decision` skill。
- 必须区分 OKX 原生 mark/index/order book 和 search-derived 证据。
- 必须执行 confidence cap、manual-only、RiskGate。
- 必须把 Bark 作为人工提醒，而不是自动交易。
- 必须能回填人工反馈和未来行情结果。

这些规则是业务核心，应该由本项目自己的 runner、risk、skill runtime 控制。

开源工具更适合放在观测、评估、展示和导出层：

```text
业务编排层：本项目自研并保持强控制
观测记录层：TraceRecorder 抽象
默认存储层：SQLite Trace Ledger
可选导出层：OpenTelemetry / Langfuse / Phoenix
离线评估层：promptfoo / 自建 replay
```

### 13.2 可集成工具清单

| 工具 | 类型 | 适合用途 | 当前是否建议接入 |
|---|---|---|---|
| OpenTelemetry GenAI | 语义标准 / trace 导出 | 统一 trace/span 字段，后续接 OTLP | 先按字段设计，不立即强接 |
| Langfuse | LLM observability / prompt / eval | UI 查看 trace、prompt 版本、eval、成本 | 后续可选 |
| Arize Phoenix | AI observability / tracing / eval | trace 展示、dataset、experiment、eval | 后续可选 |
| OpenLLMetry / Traceloop | LLM 自动 instrumentation | 自动采集 LLM 调用和 OpenTelemetry span | 后续可选 |
| promptfoo | LLM eval / regression / red team | 离线回归测试 prompt、skill、risk 输出 | Eval Dataset 和 Replay 稳定后可考虑 |
| LangGraph | agent orchestration | 状态图式 agent 编排 | 暂不替换当前 runner |
| CrewAI / AutoGen | 多 agent 框架 | 快速搭多 agent 原型 | 暂不替换当前 runner |

### 13.3 为什么不直接上 agent 框架

LangGraph、CrewAI、AutoGen 这类框架能解决“agent 如何协作”，但不能天然解决本项目最关键的生产约束：

- search-derived 证据不能替代交易所原生执行事实。
- 缺 mark/index/order book 时开仓类动作必须 hard block。
- 模型输出概率不能绕过 confidence cap。
- 任何输出都必须 manual-only。
- skill hash、prompt version、risk rule version 必须绑定到每次决策。
- 用户手动操作结果和未来行情表现需要回流复盘。

如果现在把主链路迁移到外部 agent 框架，会增加调试复杂度，并削弱我们对风控和审计边界的控制。

因此当前原则是：

```text
不替换 runner。
不把 RiskGate 交给 agent 框架。
不让外部框架决定是否下发操作提醒。
只把观测数据导出给外部工具。
```

### 13.4 推荐的可插拔架构

设计一个 `TraceRecorder` 接口，业务代码只依赖接口，不依赖具体平台。

```text
PlanRunner / Research / Risk / Notifier
  -> TraceRecorder
      -> SQLiteTraceSink        当前默认
      -> OTLPTraceSink          后续可选
      -> LangfuseTraceSink      后续可选
      -> PhoenixTraceSink       后续可选
```

接口示意：

```python
class TraceRecorder:
    def start_trace(self, **attrs): ...
    def start_span(self, trace_id: str, name: str, span_type: str, **attrs): ...
    def end_span(self, span_id: str, status: str, **attrs): ...
    def record_event(self, trace_id: str, name: str, **attrs): ...
    def record_artifact(self, trace_id: str, artifact_type: str, **attrs): ...
```

这样后续接入 Langfuse 或 Phoenix 时，只需要新增 sink，不需要重写业务编排。

### 13.5 分阶段集成计划

#### 阶段 A：只做本地 Trace Ledger

目标：

- SQLite 记录 traces、spans、events、feedback。
- CLI 查询 `trace-list`、`trace-show`。
- 不引入外部平台。

这是下一步最建议做的阶段。

#### 阶段 B：按 OpenTelemetry 语义对齐字段

目标：

- span 字段尽量贴近 OpenTelemetry GenAI 命名。
- 模型、token、latency、tool call、agent 阶段具备标准化映射。
- 仍然只写本地 SQLite。

好处是未来导出成本低。

#### 阶段 C：增加离线 eval

目标：

- 把 trace 提取成 eval case。
- 用 promptfoo 或自建 replay 比较不同 prompt / skill / risk rule 版本。
- 先做 CLI，不做 UI。

#### 阶段 D：可选接外部观测平台

触发条件：

- 本地 trace 已经稳定。
- 需要可视化查看长链路。
- 需要多人协作看 prompt、成本、latency、eval 结果。
- SQLite 查询已经不够用。

可选路径：

- 接 Langfuse：适合 prompt 管理、trace UI、eval 组合。
- 接 Phoenix：适合 OpenTelemetry tracing、dataset、experiment、eval。
- 接 OTLP collector：适合未来统一接公司级可观测平台。

### 13.6 最终建议

当前阶段只把这套开源工具集成策略记录下来，后续优先实现本地 Trace Ledger。

不要现在接 Langfuse / Phoenix / LangGraph，也不要重写 agent 编排。

先保证：

```text
每次运行有 trace_id
每个关键阶段有 span
每个错误能归因
每个高价值失败样本能进入 eval dataset
后续能无痛增加外部 sink
```

这比现在直接上现成平台更稳，也更符合本项目“交易提醒必须可控、可审计、可阻断”的边界。

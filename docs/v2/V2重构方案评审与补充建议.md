# V2 重构方案评审与补充建议

> 日期：2026-07-12
>
> 状态：建议文档，供评审参考
>
> 目的：在现有 docs/v2 设计基础上，结合官方文档深度研究和评测体系调研，识别考虑不全面之处并给出补充建议

## 一、重构动机理解

### 1.1 V1 的核心问题

V1 是一个"安全边界清晰但陷入迁移僵局"的系统。它的核心问题不是某个局部 Bug，而是架构层面的根本性矛盾：

**自研控制面与官方框架重复**。V1 不使用任何 LLM 框架（无 LangChain/LangGraph），却同时自建了 workflow executor、agent swarm、orchestration runtime、tool executor、research pipeline、telemetry 系统和多套 SQLite store。这些模块分别实现了官方框架已提供的状态机、Tool 协议、并发、重试、模型接入、Structured Output、追踪和恢复能力。

**三条竞争路径权威不清**。legacy_prompt（生产）、candidate sidecar（候选）、controlled shadow（审计）三条路径并存，`controlled_adapter.py` 硬编码 `main_action="no trade"`，Agent Swarm 永远无法接管生产。最核心的 `decision.final_input_mode` 默认仍是 `legacy_prompt`，swarm 产物 `decision_effect=none`。

**文档膨胀替代交付**。37 篇正式设计文档 + 35+ 篇迁移 checkpoint，但 hosted production、真实通知和成熟 outcome 仍未闭环。

### 1.2 重构的核心原则（用户约束）

1. **完全用框架实现**：不自己造轮子，不为了写而写一堆烂代码
2. **可维护性优先**：开发者能长期维护
3. **用户体系先预留**：先把完整 agent 流程跑通，多用户后续启用
4. **结合官方文档**：HITL、人机交互前后端设计要参考官方实践
5. **评测体系要建立**：结合 Langfuse/LangSmith 和 OneWO AgentOps 经验

## 二、V2 现有方案评估

### 2.1 亮点（应保留的设计）

**框架约束纪律性强**。02-official-framework-constraints.md 将"优先使用官方框架"转化为 CI 可执行规则，包括 import-linter 导入契约、框架所有权测试、禁止模式检查和 schema 门禁。这直接解决了 V1 "自研一切"的根本问题。

**Canonical Graph 设计清晰**。10 个节点明确区分确定性节点和 Agent 节点，Agent 负责分析和生成候选结果，确定性节点负责事实校验、风险和副作用授权。这个设计正确地把 V1 的风险规则保留为纯函数。

**数据权威分离合理**。LangGraph Checkpointer 管执行状态、Product PostgreSQL 管业务记录、LangSmith 管调试 Trace、Langfuse 管生产成本/会话，四者职责清晰。

**渐进持久化设计**。每个外部成本阶段先写完整业务 payload 再放 ID 到 State，而不是最后一个大而全的 `persist_result`。这对 Checkpoint 恢复至关重要。

**Deep Agents 隔离策略**。通过窄的 `ResearchBundle` 契约隔离 pre-1.0 依赖，限制在只读研究域，不赋予风险裁决和副作用权限。

### 2.2 不足与考虑不全面之处

#### 问题 1：过度设计，零实现验证

V2 产出了约 3500 行设计文档和 8 份 ADR，全部处于 "Proposed" 状态，但零行实现代码。在写一行代码前就锁定了 7 个产品模式、多租户、商业化等大量架构决策。核心风险是：如果 LangGraph Agent Server 的实际行为与文档预期不符，所有下游设计都需要返工。

**建议**：将 Phase 0 拆为"验证型骨架"和"产品型骨架"两步。先花 2-3 天验证 Agent Server + React SDK 的核心假设（HITL resume、断线重连、checkpoint 恢复、streaming protocol），验证通过后再审批完整设计。

#### 问题 2：架构跃迁过大

V1 仅需 SQLite 一个文件。V2 需要 PostgreSQL + Redis + Agent Server + LangSmith + Langfuse + Tavily + LLM API。对于一个"手动提醒"工具，运维复杂度显著增加。虽然 V2 的最终目标是 C 端产品，但第一阶段就应该要求这么多基础设施吗？

**建议**：Phase 0-1 可以用 `langgraph dev`（内置 in-memory checkpointer + SQLite）快速验证主链。Phase 2 再切到 PostgreSQL + Redis + Agent Server 部署。这样第一阶段只需 LLM API 和交易所 API 两个外部依赖。

#### 问题 3：评测体系设计过于粗略

V2 在 Phase 4 才提到"LangSmith Dataset、Experiment、Evaluator 和 Release Gate"，但没有详细设计评测指标体系、评测集构建方法、LLM-as-Judge 实施方案和轨迹归因分析。这与用户简历中 OneWO AgentOps 的深度不匹配。

**建议**：见下方第六节的完整评测体系设计方案。

#### 问题 4：Deep Agents 的必要性质疑

V2 仅将 Deep Agents 用于"受限研究任务"，但又必须处理其 pre-1.0 不稳定性、默认 Filesystem/Subagent/Patch 工具的安全风险。对于一个加密货币提醒系统，`create_agent` + Web Search Tool 可能就足够了。

**建议**：Phase 0-2 先用 `create_agent` + Tavily Web Search。如果研究发现需要多步规划、子代理委派和上下文管理，再在 Phase 3 引入 Deep Agents。这样避免了过早依赖 pre-1.0 包。

#### 问题 5：前端完全重建的必要性

V1 前端已有功能完整的运行列表、运行详情驾驶舱、评估工作台等页面，且有 Zod 验证 + 防御性 fallback 的安全设计。V2 要求前端完全使用 `@langchain/react` 重建，但 V1 的安全实践是有价值的。

**建议**：前端确实需要重建（因为后端协议从自定义 API 变为 Agent Server Protocol v2），但 V1 的 Zod schema 验证、防御性 fallback 和诊断门禁等安全实践应保留。重建是数据源和状态管理的重建，不是安全实践的废弃。

#### 问题 6：成本估算和降级策略缺失

V2 需要多个付费服务，但没有月度运营成本估算。对于"Agent Server 宕机"、"LangSmith 不可用"、"PostgreSQL 主库故障"等场景的降级行为不够具体。

**建议**：在 08-production-governance 中补充成本估算表和降级矩阵。

#### 问题 7：V1 已验证能力的迁移路径不清晰

V1 已跑通真实 OKX API、Bark 通知、风控规则。V2 声明"不复制 V1 结构"，但这些已验证的能力需要重新实现。文档提到"迁移规则和 golden case"，但没有说明哪些 V1 代码可以逻辑移植。

**建议**：补充 V1→V2 能力迁移矩阵，明确哪些是"逻辑移植"（风控规则、OKX 数据标准化）、哪些是"框架替换"（workflow→StateGraph、httpx→ChatOpenAI）、哪些是"完全废弃"（agent_swarm、orchestration）。

## 三、官方文档研究结论与建议

### 3.1 HITL 完整工作流程

LangGraph 的 HITL 基于 `interrupt()` + `Command(resume=...)` 两个核心 API。完整流程如下：

```
1. 前端发起请求（带 thread_id）
   -> POST /threads/{thread_id}/runs/stream

2. 后端图执行到 interrupt()
   -> GraphInterrupt 异常抛出
   -> 当前状态被 checkpoint 持久化
   -> 执行无限期暂停
   -> SSE 流返回 __interrupt__ 信息

3. 前端检测中断
   -> stream.interrupts 包含 Interrupt(value=..., id=...)
   -> stream.isLoading 变为 false
   -> 读取 interrupt.value 渲染审批/输入 UI

4. 用户操作（approve/reject/edit）

5. 前端恢复执行
   -> stream.respond({ approved: true })       // @langchain/react v2
   -> 或 stream.submit(null, { command: { resume: {...} } })  // SDK
   -> 必须使用相同的 thread_id
   -> 节点从头重新执行，interrupt() 返回 resume 值
```

**关键行为：节点重新执行**。resume 后，包含 `interrupt()` 的节点会从头重新执行，不是从 interrupt 行恢复。因此 interrupt 之前的代码必须幂等。

**V2 方案的 HITL 设计与官方一致**，文档中已正确识别了节点重执行的幂等性要求和 `respondAll()` 处理多 Interrupt 的方式。但有以下补充：

### 3.2 HITL 补充建议

**建议 1：interrupt 前的副作用拆分**。V2 文档提到"interrupt 前禁止非幂等副作用"，但应更具体地说明拆分策略。将外部调用、随机数、时间读取和数据库写入拆成独立幂等节点或使用 `@task` 装饰器。例如：

```python
# 错误：interrupt 前有副作用
def analyze_market(state):
    snapshot = fetch_market_data()  # 副作用！resume 后会重复执行
    approval = interrupt({"snapshot": snapshot, "question": "确认分析？"})
    if approval["approved"]:
        return {"decision": analyze(snapshot)}

# 正确：拆成独立节点
def collect_market_snapshot(state):
    snapshot = fetch_market_data()
    return {"market_snapshot": snapshot}  # 幂等写入 State

def confirm_analysis(state):
    approval = interrupt({"snapshot": state["market_snapshot"]})
    if approval["approved"]:
        return {"decision": analyze(state["market_snapshot"])}
```

**建议 2：三种 HITL 实现方式的选用**。官方提供三种方式：
- `interrupt()` 在节点内调用（推荐，最灵活）
- `interrupt_before` 静态断点（不推荐生产，仅调试用）
- `HumanInTheLoopMiddleware`（`create_agent` 专用，适合 Tool 级审批）

V2 应根据场景选择：分析确认用 `interrupt()`、工具级审批用 `HumanInTheLoopMiddleware`、调试用 `interrupt_before`。

**建议 3：HITL 不限于原对话页**。V2 文档已提到 `/inbox` 处理后台任务的审批，这是正确的。但要确保 `interrupt_inbox` 表的投影与 LangGraph 中断状态的一致性。建议使用 Outbox 模式：Graph interrupt 产生时在同一事务写入 `interrupt_inbox` 投影，用户响应时先更新 Graph 再更新投影。

### 3.3 Agent Server 部署建议

**Agent Server 的核心优势**是 API 服务器 + 队列工作器分离架构，提供持久化开箱即用、线程级串行化、任务队列 + 容错、SSE 流式 + 可恢复、HITL 内建支持。

**三种部署模式**：
1. Cloud (SaaS) - LangChain 管理，快速上线
2. Standalone Server - 自有基础设施，轻量级
3. Self-hosted Full Platform - 自有 K8s，企业级

**建议**：Phase 0-1 用 `langgraph dev`（本地开发模式，内置 everything）。Phase 2 用 Standalone Server（Docker Compose 部署）。Phase 5+ 根据许可和数据驻留需求决定是否切到 Cloud 或 Self-hosted。

### 3.4 前端交互建议

**`@langchain/react` v1 的 `useStream` 是前端唯一状态源**。关键 API：

```typescript
const stream = useStream({
  apiUrl: "http://localhost:2024",
  assistantId: "agent",
  threadId: threadId,
});

// 状态
stream.values     // 当前图状态
stream.messages   // 消息列表（支持 token 级流式）
stream.toolCalls  // 工具调用及结果
stream.interrupts // 中断数组
stream.isLoading  // 是否正在执行

// 操作
stream.submit()   // 发送输入
stream.stop()     // 取消执行
stream.respond()  // 恢复中断
```

**建议**：V2 前端应使用 `StreamProvider` 避免 prop drilling，支持乐观更新（`optimisticValues`）、断线重连（`reconnectOnMount`）和从 checkpoint 重新生成（`forkFrom`）。

### 3.5 create_agent 与中间件

**`create_agent` 是当前推荐 API**（`create_react_agent` 已废弃）。关键特性：
- 支持 `response_format` 结构化输出（推荐 `ProviderStrategy`，使用模型原生能力）
- 支持 `middleware` 系统（替代旧的 pre_model_hook/post_model_hook）
- 内置中间件：`HumanInTheLoopMiddleware`、`SummarizationMiddleware`、`ModelRetryMiddleware`、`ModelCallLimitMiddleware`、`PIIMiddleware` 等

**建议**：V2 的 Agent 角色中间件矩阵（02 文档 3.8 节）设计合理，但应补充中间件执行顺序的测试断言。官方规定 before hooks 正序、after hooks 逆序、wrap hooks 嵌套，V2 应在测试中固定这个顺序。

## 四、HITL 与人机交互前后端设计建议

### 4.1 后端 HITL 设计

V2 的 HITL 场景应分层设计：

**第一层：产品级 HITL（用户交互）**
- 分析确认：用户发起分析后，系统在 `build_final_result` 节点后 interrupt，展示分析结果供用户确认
- 通知确认：发送 Bark 高敏感通知前 interrupt，等待用户 approve
- 参数修正：用户要求修改模型生成的关注问题后继续

**第二层：工具级 HITL（安全审批）**
- 高风险工具调用：如果后续增加具有资金或外部写副作用的 Tool，使用 `HumanInTheLoopMiddleware`
- 管理员操作：跨租户修复或数据重放

**第三层：系统级 HITL（运维审批）**
- 配置变更审批
- 生产发布门禁

### 4.2 前端 HITL 交互设计

**中断检测与渲染**：

```tsx
// 中断时 isLoading 变为 false，interrupt 不为空
if (stream.interrupt) {
  const { type, data } = stream.interrupt.value;
  switch (type) {
    case "analysis_confirmation":
      return <AnalysisConfirmationCard data={data} />;
    case "notification_approval":
      return <NotificationApprovalCard data={data} />;
    default:
      return <GenericInterruptCard interrupt={stream.interrupt} />;
  }
}
```

**恢复响应**：

```tsx
// 单个中断
stream.respond({ approved: true, edits: {...} });

// 多个中断（同一 checkpoint）
stream.respondAll({
  [interruptA.id]: { approved: true },
  [interruptB.id]: { approved: false },
});

// 原子响应 + 状态更新
stream.respond({ approved: true }, { update: { notification_plan: newPlan } });
```

**Inbox 跨页面处理**：

用户可以在 `/inbox` 处理后台任务的 approve/edit/respond/reject。关键设计：
- `interrupt_inbox` 表保存可查询投影（interrupt_id、thread_id、type、status、created_at）
- Graph 中断产生时同一事务写入投影
- 用户响应时先通过 SDK `respond()` 恢复 Graph，Graph 恢复成功后更新投影状态
- 多设备并发使用 `(interrupt_id, checkpoint_id, response_version)` 幂等键，第一个成功响应获胜

### 4.3 流式传输与前端状态映射

| 流模式 | 用途 | 前端渲染 |
|--------|------|----------|
| `values` | 每步后完整状态快照 | 阶段进度更新 |
| `messages-tuple` | token 级流式 | 逐字渲染模型分析 |
| `updates` | 每节点增量更新 | 节点完成动画 |
| `custom` | 节点内自定义事件 | 进度条、证据收集进度 |
| `tools` | 工具执行进度 | Tool card 状态切换 |

**建议**：V2 的 custom channel 设计（`custom:task_progress`、`custom:artifact`、`custom:evidence` 等）合理，但应确保每个 custom channel 有 typed schema、版本号、owner 和明确的 UI consumer。

## 五、用户体系预留建议

### 5.1 第一阶段：固定开发身份

V2 方案已设计了 `AUTH_MODE=development` 的固定开发身份模式，这是正确的。补充建议：

**建议 1：ActorContext 从第一天传播**。虽然不启用正式鉴权，但所有业务表和 Graph Thread/Run Metadata 从第一天就携带 `tenant_id` 和 `user_id`。这避免了后续迁移成本。

**建议 2：开发身份安全边界**。V2 已正确设计了 loopback 绑定、environment 检测和权限限制。补充建议在 CI 中加入"开发身份不能在生产构建中启用"的强制检查。

### 5.2 后续阶段：正式鉴权

V2 推荐使用 Agent Server 的 `Auth` authenticate/authorization handlers。关键设计：
- Thread 是 Run 的授权父资源
- Store namespace 强制改写为 `(tenant_id, user_id, purpose, ...)`
- Custom routes 在 auth-first 后显式调用 `IdentityProvider.authorize()`

**建议**：正式 Auth Provider 推荐 Auth.js（与 Next.js 生态一致）。但 Phase 5 之前不实现，只预留接口和数据字段。

## 六、评测体系设计方案

### 6.1 设计目标

结合用户简历中 OneWO AgentOps 的经验，评测体系需要实现：

1. **评测集与黄金标准**：1000+ 条离线评测集，覆盖意图识别、任务规划、工具选择、多轮补全与风险拦截
2. **自动评审**：LLM-as-Judge 初评 + 人工抽检校准
3. **版本回归与轨迹归因**：对 Prompt、模型、RAG 和 Tool Schema 执行同集回归，结合 LangSmith 轨迹定位问题环节
4. **优化结果回流**：评测结论回流至 Prompt、Few-shot、工具描述与路由策略

### 6.2 技术选型

**推荐方案：Langfuse（自部署）+ DeepEval + RAGAS**

| 组件 | 工具 | 用途 | 选择理由 |
|------|------|------|----------|
| 可观测性 + 轨迹追踪 | Langfuse（自部署） | Trace 记录、Session 追踪、在线评测 | MIT 开源、免费自部署、数据主权 |
| 离线评测框架 | DeepEval | Agent 指标体系（8+3 指标）、自动化测试、CI/CD | Apache-2.0、唯一完整 Agent 评测指标体系 |
| RAG 专项评测 | RAGAS | Faithfulness/Context Recall 等 | RAG 评测事实标准 |
| LLM-as-Judge | DeepEval G-Eval | CoT 评判 + 概率加权评分 | 与人类评分 Spearman 0.514 |
| 人工标注 | Langfuse Annotation Queues | 人工抽检校准 | 内置工作流 |
| 回归测试 | Langfuse CI/CD Experiments | 阻止回归部署 | 原生支持 |

**备选方案**：如果使用 LangChain 生态且预算允许，可直接用 LangSmith（Plus $39/seat/月，50K traces），其轨迹归因能力最为成熟，与简历描述的技术栈完全吻合。

**建议**：本项目使用 LangGraph 框架，LangSmith 自动追踪开箱即用。建议 LangSmith（Trace/调试/Dataset/回归）+ Langfuse（生产成本/会话/自部署）双路接入，与 V2 现有设计一致。离线评测框架使用 DeepEval 补充 LangSmith 不具备的 Agent 专用指标体系。

### 6.3 评测指标体系

#### 分层架构

```
┌─────────────────────────────────────────────────┐
│            整体指标（业务层）                      │
│  任务完成率 | 端到端耗时 | 风险拦截率 | 用户满意度  │
├─────────────────────────────────────────────────┤
│            轨迹指标（流程层）                      │
│  平均步数 | 步数效率 | 多轮补全率 | 死循环率        │
├─────────────────────────────────────────────────┤
│            分环节指标（组件层）- 归因分析核心       │
│  意图准确率 | 工具选择正确率 | 参数准确率           │
│  召回率 | 路由准确率 | 生成忠实度 | 拒答准确率       │
├─────────────────────────────────────────────────┤
│            基础设施指标（系统层）                  │
│  API成功率 | 延迟P50/P99 | Token消耗 | 错误率      │
└─────────────────────────────────────────────────┘
```

#### 本项目专用指标

| 指标 | 定义 | 测量方法 | 目标 |
|------|------|----------|------|
| 分析准确率 | 方向判断与 matured outcome 的一致性 | hit_rate + Brier score | ≥ 60% hit rate |
| 风险拦截率 | 高风险操作正确拦截比率 | 正确拦截数 / 高风险请求总数 + 误拦截率 | ≥ 95% 拦截，≤ 5% 误拦截 |
| 证据覆盖率 | 分析引用的证据占可用证据的比例 | 引用证据数 / 可用证据数 | ≥ 80% |
| 数据新鲜度 | 行情数据从交易所到分析的时间差 | analysis_time - data_fetch_time | ≤ 90 秒 |
| 通知到达率 | Bark 通知成功到达的比率 | sent / attempted | ≥ 99% |
| 风控规则命中率 | 各风控规则被触发的频率和分布 | 按 rule_id 统计 | 监控异常模式 |

### 6.4 评测集构建方法

**数据来源三通道**：

1. **线上脱敏 Query（50%）**：通过 LangSmith 的 `convert_runs_to_test()` 将生产 Trace 自动转化为评测样本，脱敏处理
2. **Bad Case 收集（20%）**：从用户反馈（点踩/投诉）、人工发现的错误案例中提取
3. **人工补充样本（30%）**：针对覆盖维度人工构造

**覆盖维度**：

| 维度 | 占比 | 场景示例 |
|------|------|----------|
| 意图识别 | 15% | 分析请求、追问、比较、监控设置 |
| 市场分析 | 25% | BTC/ETH/SOL，不同周期，牛市/熊市/震荡 |
| 证据评估 | 15% | 证据充足、证据缺失、来源冲突、过期数据 |
| 风险拦截 | 20% | 高杠杆、不允许交易对、auto_order、数据过期 |
| 降级处理 | 10% | OKX 不可用、Web Search 失败、模型超时 |
| 边界测试 | 15% | 空输入、超长 query、不支持的标的、极端行情 |

### 6.5 LLM-as-Judge 实施方案

**评判模型**：使用 GPT-4 或 Claude（与生产模型不同，避免 self-enhancement bias），temperature=0

**评判维度**（每条样本 4 维度评分 1-5 分）：
1. 意图准确性：是否正确理解用户意图
2. 工具选择正确性：是否选择了正确的工具和参数
3. 任务完成度：是否完整完成了任务
4. 回答质量：分析是否专业、有逻辑、可执行

**CoT 评判**（G-Eval 方法）：
```
Step 1: 让 Judge 模型先生成评判推理（CoT）
Step 2: 基于推理给出 1-5 分评分
Step 3: 利用 token 概率计算加权分数：score = Σ(score_i × P(score_i))
```

**人工抽检校准**：
- 建立 Golden Set（150-300 条人工标注样本）
- 分层抽样：高置信度抽检 5-10%，低置信度抽检 50-100%
- 一致性目标：LLM 与人工一致率 ≥ 75%
- 每月重新校准，模型更新时立即重新校准

### 6.6 版本回归与轨迹归因

**回归触发场景**：Prompt 变更 / 模型升级 / RAG 策略调整 / Tool Schema 修改 / 风控规则变更

**回归流程**：
1. 版本变更触发自动回归测试（CI/CD pipeline）
2. 在固定版本评测集上运行新版本
3. 收集各层指标，与基线版本对比
4. 三级判定：
   - **硬回归**（完成→失败，阻断发布）
   - **软回归**（指标下降超阈值，人工 Review）
   - **趋势回归**（连续下降，长期监控）
5. 对回归失败案例，通过 LangSmith/Langfuse Trace 逐层展开定位问题环节

**轨迹归因分析**：

利用 LangSmith/Langfuse 的 Trace 能力，对每个失败 case 展开 Trace：

| 失败类型 | Trace 定位方法 | 优化方向 |
|----------|---------------|----------|
| 召回问题 | retriever 节点展示 query 和返回 documents | 优化检索策略/分块/Embedding |
| 路由问题 | LLM 节点展示 tool_calls 决策 | 优化路由 Prompt/阈值 |
| 工具调用问题 | tool 节点展示输入参数和返回结果 | 优化 Tool Description/参数校验 |
| 生成问题 | 最终 LLM 节点展示完整 prompt 和 completion | 优化生成 Prompt/Few-shot |
| 风控问题 | risk_gate 节点展示规则命中 | 优化风控规则/阈值 |

### 6.7 优化结果回流

评测结论回流至以下优化点：

| 优化方向 | 触发条件 | 具体操作 |
|----------|----------|----------|
| Prompt 优化 | 失败 case 归因分析 | 修改对应环节的 Prompt |
| Few-shot 补充 | 高频失败 case | 作为 Few-shot 示例加入 Prompt |
| 工具描述优化 | 工具选择错误率高 | 优化 Tool Description |
| 路由策略调整 | 意图混淆矩阵显示混淆模式 | 调整路由阈值或增加路由规则 |
| 风控规则优化 | 误拦截或漏拦截 | 调整风控规则参数 |
| 评测集更新 | 新发现的 Bad Case | 加入评测集，防止回归 |

### 6.8 评测体系实施路线图

| 阶段 | 评测能力 | 对应 V2 Phase |
|------|----------|---------------|

## 七、渐进式实施建议

### 7.1 修订后的 Phase 0：验证型骨架

**目标**：用最小成本验证核心假设，不审批完整设计

**时间**：2-3 天

**交付**：
- `langgraph dev` 本地启动
- 一个最简 `StateGraph`（3 节点：input → create_agent → output）
- 一个 `useStream` 页面
- 验证 HITL interrupt/resume
- 验证刷新重连和 checkpoint 恢复
- LangSmith Trace 可见

**退出条件**：
- interrupt 后刷新页面，仍可恢复
- Agent Server 重启后 Thread 可恢复
- 无自定义 SSE、Thread Store 或 LLM Client

**意义**：验证通过后，才有信心审批 3500 行设计文档。如果发现官方行为与预期不符，可以在损失最小的时候调整设计。

### 7.2 修订后的 Phase 1：真实主链（最小基础设施）

**目标**：跑通真实市场分析闭环，仍用 `langgraph dev`

**交付**：
- OKX 行情 Tool（逻辑移植 V1 的 provider）
- Tavily Web Search Tool
- `create_agent` + Structured Output
- 纯函数 Evidence Gate 和 Risk Gate（逻辑移植 V1）
- SQLite 产品表（最小可用，不引入 PostgreSQL）
- Bark 通知
- 前端业务组件展示行情、证据、模型结论、风险门禁

**关键决策**：此阶段不引入 PostgreSQL/Redis/Agent Server 部署。用 `langgraph dev` 内置的 in-memory checkpointer + SQLite 产品表。这样第一阶段只需 LLM API + OKX API + Tavily API 三个外部依赖。

**退出条件**：
- 一个真实 Run 从浏览器到模型、行情、搜索、风险、数据库和通知
- 普通页面无原始 JSON
- Playwright 覆盖 success/blocked/failed

### 7.3 修订后的 Phase 2：部署升级 + Agent UX

**目标**：从 `langgraph dev` 切到 Agent Server 部署，补全 Agent 工作空间

**交付**：
- PostgreSQL + Redis + Agent Server（Docker Compose）
- PostgresSaver Checkpointer
- 数据迁移：SQLite → PostgreSQL
- 完整前端工作空间（Home、Work、Inbox、Library）
- HITL interrupt/resume 完整交互
- Tool card、Evidence timeline、Risk panel
- Langfuse 接入

**退出条件**：
- Agent Server 重启后 Thread/Checkpoint 可恢复
- Interrupt 刷新后仍可处理
- 断线重连无重复内容

### 7.4 后续 Phase 保持 V2 原设计

Phase 3-6 保持 V2 原有路线图，但补充评测体系实施路线图（见 6.8 节）。

## 十、总结

V2 的设计文档质量很高，对官方框架的调研非常深入，反重造纪律和渐进式交付理念都值得肯定。核心建议如下：

1. **先验证后审批**：Phase 0 拆为验证型骨架（2-3 天），验证核心假设后再审批完整设计
2. **渐进式基础设施**：Phase 0-1 用 `langgraph dev` + SQLite，Phase 2 再引入 PostgreSQL + Redis + Agent Server
3. **Deep Agents 延后**：Phase 0-2 用 `create_agent` + Tavily，Phase 3 按需引入 Deep Agents
4. **评测体系尽早启动**：Phase 1 就建 100 条基础评测集，Phase 4 扩展到 1000+ 条完整体系
5. **评测技术选型**：LangSmith（Trace/调试/Dataset/回归）+ Langfuse（生产成本/自部署）+ DeepEval（Agent 专用指标）+ RAGAS（RAG 专项）
6. **V1 能力迁移矩阵**：明确哪些逻辑移植、哪些框架替换、哪些完全废弃
7. **前端保留安全实践**：Zod 验证、防御性 fallback、诊断门禁等 V1 安全实践应在重建中保留

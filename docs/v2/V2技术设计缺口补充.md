## 一、Graph 拓扑与并发设计

### 1.1 完整拓扑定义

V2 文档列出了 10 个节点但未给出实际边定义。补充如下：

```python
from langgraph.graph import StateGraph, START, END
from langgraph.types import Send

def build_graph() -> "CompiledGraph":
    builder = StateGraph(AnalysisState)

    # 注册节点
    builder.add_node("bootstrap_run", bootstrap_run)
    builder.add_node("validate_request", validate_request)
    builder.add_node("collect_market_snapshot", collect_market_snapshot)
    builder.add_node("research_events", research_events)
    builder.add_node("analyze_market", analyze_market)
    builder.add_node("validate_evidence", validate_evidence)
    builder.add_node("apply_risk_policy", apply_risk_policy)
    builder.add_node("build_final_result", build_final_result)
    builder.add_node("confirm_analysis", confirm_analysis)  # HITL interrupt
    builder.add_node("commit_final_artifact", commit_final_artifact)
    builder.add_node("complete_run", complete_run)

    # 线性边
    builder.add_edge(START, "bootstrap_run")
    builder.add_edge("bootstrap_run", "validate_request")

    # validate_request 后条件路由
    builder.add_conditional_edges(
        "validate_request",
        route_after_validation,  # 返回 "parallel_collect" 或 "blocked"
    )

    # collect_market_snapshot 和 research_events 并行
    # LangGraph 支持同一节点后添加多条边来实现并行
    builder.add_edge("collect_market_snapshot", "analyze_market")
    builder.add_edge("research_events", "analyze_market")
    # analyze_market 会等待两个并行节点都完成后执行（barrier 语义）

    builder.add_edge("analyze_market", "validate_evidence")
    builder.add_edge("validate_evidence", "apply_risk_policy")

    # apply_risk_policy 后条件路由
    builder.add_conditional_edges(
        "apply_risk_policy",
        route_after_risk,  # 返回 "build_final_result" 或 "commit_blocked"
    )

    builder.add_edge("build_final_result", "confirm_analysis")

    # confirm_analysis 后条件路由（HITL 响应）
    builder.add_conditional_edges(
        "confirm_analysis",
        route_after_confirm,  # 返回 "commit_final_artifact" 或 "build_final_result"(编辑后重分析) 或 "commit_blocked"
    )

    builder.add_edge("commit_final_artifact", "complete_run")
    builder.add_edge("complete_run", END)

    # 导出编译后的 graph（不配 checkpointer，Agent Server 自动注入）
    return builder.compile()
```

### 1.2 并行设计决策

**`collect_market_snapshot` 和 `research_events` 并行执行**。理由：

- 数据依赖独立：行情采集只依赖 `validate_request` 的输出，研究只依赖 `validate_request` 的输出
- 性能提升显著：串行约 40s（行情 10s + 研究 30s），并行约 30s（取较慢者）
- LangGraph 原生支持：同一节点后添加多条边即并行，到达汇聚节点时自动 barrier

**实现方式**：LangGraph 的并行是通过 add_edge 多条边实现的，不是 Send。Send 用于动态 fan-out（如同一节点需要为 N 个不同输入并行执行），这里两个节点职责不同，用多边并行即可。

### 1.3 条件路由函数

```python
def route_after_validation(state: AnalysisState) -> str:
    if state["errors"]:
        return "commit_blocked"
    return "parallel_collect"

def route_after_risk(state: AnalysisState) -> str:
    risk = state["risk_verdict"]
    if risk and risk.blocked:
        return "commit_blocked"
    return "build_final_result"

def route_after_confirm(state: AnalysisState) -> str:
    # confirm_analysis 节点内 interrupt() 的返回值决定路由
    approval = state.get("approval_result", {})
    if approval.get("action") == "approve":
        return "commit_final_artifact"
    elif approval.get("action") == "edit":
        return "build_final_result"  # 重新构建结果
    else:  # reject
        return "commit_blocked"
```

## 二、Graph 限制与预算

### 2.1 限制值定义

| 限制维度 | 值 | 实现方式 | 说明 |
|----------|-----|----------|------|
| Graph 总执行时间 | 180s 硬限 | Agent Server `timeout` 配置 | 超时自动 cancel Run |
| p95 目标 | 120s (GA) | SLO 监控 | 含行情+研究+分析+风控 |
| recursion_limit | 30 | `config={"recursion_limit": 30}` | 默认 25，V2 设 30 留余量 |
| 模型调用次数 | 6 | `ModelCallLimitMiddleware(max_calls=6)` | 分析 1 + 研究子代理 3 + 校验 1 + 重试余量 1 |
| 工具调用次数 | 15 | `ToolCallLimitMiddleware(max_calls=15)` | 行情 3 + 搜索 5 + 研究读取 5 + 其他 2 |
| 搜索调用次数 | 8 | Deep Agents budget | 每个子代理最多 2-3 次搜索 |
| 单节点超时 | 60s | `RetryPolicy` + Provider timeout | 超时触发 RetryPolicy |
| 模型单次超时 | 30s | `ChatOpenAI(timeout=30)` | 单次 LLM 调用超时 |
| OKX API 单次超时 | 10s | httpx `timeout=10` | 单个 endpoint 超时 |
| Web Search 单次超时 | 15s | Tavily tool 配置 | 单次搜索超时 |

| 组件 | 最大重试 | 单次超时 | 总超时 | 重试策略 |

## 三、OKX API 限频与缓存策略

### 3.1 限频设计

OKX 公开 API 限制：每个 endpoint 20 req/2s。V2 的 `fetch_market_data` tool 一次调用 7 个 endpoint，需要控制频率。

**方案：Tool 内部并发 + 信号量限频**

```python
import asyncio
import httpx
from langchain_core.tools import tool

# 全局信号量：限制同时发出的 OKX 请求数
_okx_semaphore = asyncio.Semaphore(5)  # 最多 5 个并发请求

@tool
async def fetch_market_data(symbol: str, data_types: list[str]) -> dict:
    """获取交易所原生市场数据（OKX 公开 API）。"""
    async with httpx.AsyncClient(timeout=10) as client:
        tasks = []
        for data_type in data_types:
            tasks.append(_fetch_one(client, symbol, data_type))
        results = await asyncio.gather(*tasks, return_exceptions=True)
    return _assemble_snapshot(symbol, results)

async def _fetch_one(client, symbol, data_type):
    async with _okx_semaphore:  # 限频
        try:
            resp = await client.get(f"{OKX_BASE_URL}/{_endpoint(data_type)}",
                                     params={"instId": symbol})
            resp.raise_for_status()
            return data_type, resp.json(), datetime.utcnow()
        except Exception as e:
            return data_type, None, datetime.utcnow()
```

### 3.2 Redis 缓存策略

同一标的在短时间内多次分析时（如定时监控 + 用户手动触发），缓存行情数据减少 OKX API 调用。

**缓存规则**：

| 数据类型 | TTL | 缓存键 | 说明 |
|----------|-----|--------|------|
| ticker | 5s | `market:ticker:{symbol}` | 价格变化快，缓存极短 |
| mark/index | 10s | `market:mark:{symbol}` | 标记价变化较慢 |
| funding_rate | 60s | `market:fr:{symbol}` | 8h 结算周期，缓存 60s 足够 |
| open_interest | 30s | `market:oi:{symbol}` | OI 变化适中 |
| order_book | 5s | `market:ob:{symbol}` | 盘口变化极快 |
| candles | 60s | `market:candles:{symbol}:{bar}` | K线按周期缓存 |
| BTC 方向锚 | 10s | `market:btc_anchor` | 全局共享，多个分析复用 |

**关键设计**：
- 缓存只用于减少 API 调用，**不用于替代新鲜度检查**。风控的新鲜度检查基于数据时间戳，不基于缓存命中。
- 缓存 miss 时直接调用 API，不等待。
- 分析结果中的 `data_fetched_at` 始终是实际获取时间，不是缓存写入时间。

## 七、跨页面状态同步机制

### 7.1 同步策略

用户在 Inbox 确认 Interrupt 后，Work 页面需要同步更新。机制如下：

**场景 1：Work 页面有 active stream**

`useStream` 自动监听 Thread 的 event stream。当 Inbox 通过 SDK `respond()` 恢复 Graph 后，Graph 继续执行并产生新事件。Work 页面的 `useStream` 自动接收这些事件并更新 UI。**无需额外机制**。

**场景 2：Work 页面 disconnected**

用户在 Work 页面断开连接（切换标签页/网络断开），在 Inbox 确认 Interrupt。Graph 继续执行完成。用户回到 Work 页面时：

1. `useStream({ reconnectOnMount: true })` 自动重新附着 Thread
2. SDK 从 `since` sequence 开始 replay missed events
3. 如果 replay window 已过期，先从 Product `product_event_projections` API 重建稳定历史
4. 再接续 live stream

**场景 3：多标签页同时打开**

两个标签页都打开同一 Thread。在一个标签页 respond 后：
- 另一个标签页的 `useStream` 自动收到 Graph 继续执行的事件
- 如果另一个标签页也尝试 respond，幂等键 `(interrupt_id, checkpoint_id, response_version)` 保证只有第一个生效
- 第二个 respond 返回 `INTERRUPT_ALREADY_RESOLVED`，前端刷新状态

### 7.2 Inbox 到 Work 的通知

如果 Work 页面没有 active stream（用户在 Inbox 页面操作），不需要实时通知 Work 页面。用户回到 Work 页面时，`reconnectOnMount` 会自动同步。

如果需要跨页面实时通知（如"你在 Inbox 有新的待处理项"），使用 Agent Server 的 Thread-level stream：`GET /threads/{thread_id}/stream` 监听所有 Run 的事件。但这不是必须的--Inbox badge 可以通过轮询 `GET /app/inbox/{user_id}?status=pending` 实现。

## 九、日志与告警设计

### 9.1 应用日志

**日志框架**：structlog（结构化 JSON 日志）

**日志配置**：

```python
import structlog

structlog.configure(
    processors=[
        structlog.contextvars.merge_contextvars,
        structlog.processors.add_log_level,
        structlog.processors.TimeStamper(fmt="iso"),
        structlog.processors.JSONRenderer(),
    ],
)

# 每次请求注入 correlation ID
def bind_request_context(request):
    structlog.contextvars.bind_contextvars(
        request_id=request.headers.get("x-request-id", str(uuid.uuid4())),
        tenant_id=actor.tenant_id,
        user_id=actor.user_id,
        thread_id=thread_id,
        run_id=run_id,
    )
```

**日志级别规范**：

| 级别 | 使用场景 |
|------|----------|
| ERROR | 系统异常（DB 连接失败、Agent Server 不可用、未捕获异常） |
| WARNING | 业务降级（数据缺失、搜索失败、通知失败、风控阻断） |
| INFO | 正常业务事件（Run 开始、阶段完成、Run 完成、HITL 中断） |
| DEBUG | 调试信息（LLM 输入/输出摘要、Tool 调用参数、State 变化） |

**禁止记录**：API Key、Bark Key、Authorization、Cookie、完整 Prompt/Response（用 LangSmith trace 代替）。

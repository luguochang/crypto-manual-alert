## 一、产品核心价值回顾

### 1.1 这个产品到底解决什么问题

加密货币合约交易者面临三个核心痛点：

1. **分析瘫痪**：需要同时跟踪宏观（美联储/通胀/地缘政治）、衍生品（资金费率/持仓量/清算）、技术面和事件日历，信息过载导致无法及时决策
2. **情绪化交易**：缺乏纪律性框架，容易在情绪驱动下追涨杀跌
3. **风控缺位**：没有系统性的风险控制，容易过度杠杆或忽略关键数据缺失

系统通过 **11 因子结构化分析框架 + 14 条硬性风控规则 + 90 秒计划有效期 + 强制人工执行**，在关键时机推送可执行的操作提醒到用户手机。

### 1.2 核心业务闭环

```
用户发起分析（手动/定时）
  -> 拉取 OKX 实时行情（7类数据）
  -> 可选 Web Research（事件、宏观背景）
  -> 注入 crypto-macro-decision 方法论（11因子分析框架）
  -> LLM 结构化分析（输出唯一动作 + 价位 + 概率 + 反向理由）
  -> 确定性风控门禁（14条规则）
  -> Bark 推送通知到手机
  -> 用户手动打开 OKX 执行
  -> 成熟窗口后追踪 outcome（方向命中率/Brier Score/PnL）
```

### 1.3 V1 的产品级问题（非技术框架问题）

从产品业务角度，V1 存在以下**产品级**缺陷（不是技术框架问题）：

| 缺陷 | 影响 | 严重度 |
|------|------|--------|
| 用户持仓感知缺失 | hold/close/flip 动作无意义，LLM 不知道用户当前仓位 | 严重 |
| 宏观事件日历默认关闭 | 开仓类动作默认被阻断，系统无法产生可执行建议 | 严重 |
| 衍生品数据严重不足 | 缺 long/short ratio/CVD/清算热力图，置信度被限制在 58% | 严重 |
| 无多资产并行分析 | BTC 方向锚缺失，ETH/SOL 分析缺乏跨资产参照 | 中等 |
| 通知单向无交互 | 用户无法从通知操作（确认/拒绝/重新分析） | 中等 |
| 无用户面向的历史回顾 | outcome 追踪在 eval 旁路系统，用户看不到 | 中等 |
| 全局配置无个性化 | max_leverage/risk_pct 写死，用户无法调整 | 中等 |
| 无市场体制自动判断 | VIX/DXY/国债收益率等宏观数据不获取 | 中等 |

## 二、V2 设计文档缺口清单

### 2.1 产品行为规格缺失

V2 的 01 和 06 文档描述了架构和产品模式，但缺少**精确到用户操作的产品行为规格**。AI 实现时需要知道：每个页面上的每个操作，触发什么后端行为，返回什么 UI 状态，错误时显示什么。

**缺口清单**：

| 编号 | 缺口 | 影响 |
|------|------|------|
| G-01 | 无"发起分析"的精确交互规格 | AI 不知道前端表单提交后如何与 Agent Server 交互 |
| G-02 | 无分析结果的渐进式展示规格 | AI 不知道流式数据如何映射到 UI 组件 |
| G-03 | 无 HITL 确认/拒绝的 UI 交互规格 | AI 不知道中断时用户看到什么、怎么操作 |
| G-04 | 无错误和降级的用户提示规格 | AI 不知道 OKX 不可用/模型超时/搜索失败时用户看到什么 |
| G-05 | 无用户首次激活的 onboarding 流程 | AI 不知道新用户第一次打开产品的流程 |
| G-06 | 无 Watchlist 和 Monitor 的交互规格 | AI 不知道用户如何添加关注、设置监控 |
| G-07 | 无 Outcome Review 的用户界面规格 | AI 不知道用户如何查看历史决策准确性 |
| G-08 | 无通知内容和格式规格 | AI 不知道 Bark/App 通知的具体内容结构 |
| G-09 | 无移动端交互规格 | AI 不知道移动端的导航、手势、布局 |
| G-10 | 无用户偏好和风险配置规格 | AI 不知道用户如何设置自己的风险参数 |

### 2.2 Skill 方法论迁移方案缺失

V1 的 `crypto-macro-decision` skill 是系统的"大脑"，定义了 11 因子分析框架、8 步工作流、决策阶梯、事件压缩矩阵等。V2 需要明确：

- 这个 skill 的内容如何映射到 `create_agent` 的 `system_prompt`
- 11 因子评分如何通过 Structured Output 输出
- 根因链分析如何在 Graph 节点中体现
- 事件压缩矩阵如何与宏观事件日历集成
- `references/` 下的 7 个参考文档如何注入 Agent 上下文

**建议**：补充一份"Skill 方法论迁移方案"文档。

### 2.3 市场数据提供商设计缺失

V1 只对接 OKX 公开 API，但 skill 方法论要求的衍生品数据（long/short ratio、CVD、清算热力图、basis、options IV/skew）OKX 公开 API 不提供。V2 需要明确：

- 是否集成第三方数据源（CoinGlass/Velo/Deribit）
- 如何在 LangChain Tool 中封装这些数据源
- 数据源不可用时的降级策略
- 多资产并行获取（BTC 方向锚）

**建议**：补充"市场数据提供商设计"文档。

### 2.4 通知系统设计缺失

- 通知内容格式（标题/正文/操作按钮）

**建议**：补充"通知系统设计"文档。

### 2.5 用户持仓上下文设计缺失

V2 蓝图提到"portfolio contexts"但只说"用户手动录入或只读导入的持仓上下文，不含交易权限"。需要明确：

- 用户如何录入持仓（手动表单/导入/API 只读同步）
- 持仓数据如何注入 Graph State 和 Agent prompt
- 持仓变化如何更新（用户手动修改/定时刷新）
- 持仓数据的安全和隐私边界

**建议**：补充"用户持仓上下文设计"文档。

### 2.6 评测与产品的关系缺失

- 用户反馈如何进入评测数据集

## 三、产品行为规格补充设计

### 3.1 G-01：发起分析交互规格

**桌面端 Work 页面三栏布局**：

```
┌─────────────┬──────────────────────────┬──────────────────┐
│  Thread/    │  对话时间线               │  Artifact/       │
│  Task 列表  │                          │  Evidence/       │
│             │  [用户] 分析 BTC 4h      │  Risk Inspector  │
│  > BTC 4h   │                          │                  │
│    ETH 1h   │  [Agent] 正在采集行情... │  ┌────────────┐  │
│    SOL 12h  │  📊 行情采集 ✓           │  │ 行情快照    │  │
│             │  🔍 研究事件...          │  │ BTC: $67200 │  │
│  [+ 新建]   │  📝 分析中...            │  │ 资金费率: + │  │
│             │                          │  │ 持仓量: ↑   │  │
│             │  [Agent] 分析完成        │  └────────────┘  │
│             │  ┌─────────────────┐    │                  │
│             │  │ 📋 分析结论      │    │  ┌────────────┐  │
│             │  │ 方向: 做多      │    │  │ 风险门禁    │  │
│             │  │ 入场: $67100    │    │  │ ✓ 手动执行  │  │
│             │  │ 止损: $66500    │    │  │ ✓ 杠杆≤2x   │  │
│             │  │ 目标: $68500    │    │  │ ✓ 风险≤25%  │  │
│             │  │ 概率: 62%       │    │  └────────────┘  │
│             │  │                  │    │                  │
│             │  │ [确认] [拒绝]    │    │  ┌────────────┐  │
│             │  │ [编辑后确认]     │    │  │ 证据        │  │
│             │  └─────────────────┘    │  │ • ETF流入   │  │
│             │                          │  │ • 资金费率   │  │
│             │                          │  │ • OI变化    │  │
│             │                          │  └────────────┘  │
└─────────────┴──────────────────────────┴──────────────────┘
```

**交互流程**：

1. 用户在中间栏输入消息或点击"发起分析"
2. 选择交易标的（BTC/ETH/SOL）、分析周期（1h/4h/12h/24h）、可选关注问题
3. 点击提交后：
   - UI 立即 optimistic 显示用户消息
   - Agent 开始执行，中间栏显示阶段进度（使用 `custom:task_progress` channel）
   - 右栏渐进式展示行情快照、证据、风险门禁
4. 分析完成后：
   - 中间栏展示结构化分析结论卡片（方向/入场/止损/目标/概率/反向理由）
   - 卡片底部有 [确认] [拒绝] [编辑后确认] 三个操作按钮
   - 右栏展示完整证据和风险门禁详情
5. 用户点击 [确认]：
   - 触发 `stream.respond({ action: "approve" })`
   - Graph 继续 commit_final_artifact 和 complete_run
   - 如果配置了通知，发送 Bark/App 通知
   - UI 显示"已确认，请手动执行"状态
6. 用户点击 [拒绝]：
   - 触发 `stream.respond({ action: "reject" })`
   - Graph 标记为 blocked（user_rejected）
   - UI 显示"已拒绝"状态
7. 用户点击 [编辑后确认]：
   - 展开编辑面板，用户可修改入场价/止损/目标
   - 提交后触发 `stream.respond({ action: "edit", edits: {...} })`

**对应框架 API**：
- 前端：`useStream({ apiUrl, assistantId, threadId })` + `stream.submit()` + `stream.respond()`
- 后端：`interrupt()` 在 `build_final_result` 后暂停 + `Command(resume=...)` 恢复
- 渐进展示：`custom:task_progress` + `custom:evidence` + `custom:artifact` channels

### 3.2 G-04：错误和降级的用户提示规格

| 错误场景 | UI 提示 | 技术状态 |
|----------|---------|----------|
| OKX API 不可用 | "行情数据暂时不可用，分析已暂停。系统将在数据恢复后自动重试。" | `market_snapshot = null`, Run 状态 = `failed` |
| OKX 部分数据缺失 | 右栏行情面板标注"⚠️ 部分数据缺失"，列出缺失项 | `warnings` 追加 |
| Web Search 失败 | "网络研究暂时不可用，分析将基于交易所数据继续。" | `research_bundle = null`, 降级分析 |
| 模型超时 | "分析服务响应超时，请稍后重试。" + [重试] 按钮 | Run 状态 = `failed`, error_type = `ModelTemporaryError` |
| 模型返回格式错误 | "分析结果格式异常，已自动重试。如持续出现请联系管理员。" | Structured Output 自动重试 |
| 风控阻断（数据过期） | "行情数据已过期（超过90秒），建议重新发起分析。" + [重新分析] 按钮 | Run 状态 = `blocked`, rule = `market.data.stale` |
| 风控阻断（数据不完整） | "核心执行数据不完整，无法生成可执行建议。" + 展示缺失数据 | Run 状态 = `blocked`, rule = `market.core_execution.missing` |
| 通知发送失败 | "通知发送失败，但分析结果已保存。可在此页面查看。" | `notification.status = failed` |
| Agent Server 不可用 | "服务暂时不可用，请稍后刷新。" + 顶部 Banner | `stream.error` != null |
| 断线重连中 | 顶部 Banner "正在重新连接..." | `isThreadLoading = true` |
| 断线但后台继续 | 顶部 Banner "实时连接不可用，任务可能在后台运行。" + [查看任务状态] | `stream.error` + Product Task API 查询 |

**设计原则**：
- 永远不说"出错了"，而是说"什么不可用"和"建议怎么做"
- blocked 和 failed 是不同状态，blocked 是业务正常（风控起作用），failed 是技术异常
- 每个错误都有对应的用户操作入口（重试/重新分析/查看详情/联系管理员）

## 四、Skill 方法论迁移方案

### 4.1 迁移策略

V1 的 `crypto-macro-decision` skill 包含 1 个主文件（SKILL.md）和 7 个参考文档。迁移到 V2 的策略是：

| V1 组件 | V2 实现 | 说明 |
|---------|---------|------|
| SKILL.md（8步工作流 + 分析框架） | `create_agent` 的 `system_prompt` | 核心分析逻辑 |
| factors-and-sop.md（11因子评分） | `response_format` 的 Pydantic schema | 结构化输出因子评分 |
| event-pool.md（事件池） | Web Research + 数据库事件表 | 动态获取而非静态 |
| exchange-derivatives.md | `tools/market.py` 的 Tool 文档 | 指导 Agent 解读数据 |
| indicator-sweep.md | system_prompt 的指标扫描指令 | 指导 Agent 执行 |
| templates.md | system_prompt 的输出模板 | 指导 Agent 格式化 |
| lessons.md | system_prompt 的持久教训 | 过程记忆 |
| data-sources.md | Tool 的描述和优先级 | 指导 Agent 数据获取 |

### 4.2 system_prompt 结构

```python
SYSTEM_PROMPT = """
你是一个宏观aware的加密货币合约交易决策助手。你的职责是：
1. 从实时数据重建当前市场视图
2. 执行 11 因子结构化分析
3. 输出唯一可执行动作
4. 进行对抗性审查
5. 永远不自动下单，只提供人工确认建议

## 核心原则
- 分离事实、推理和执行
- 每次决策从最新数据重建，不依赖历史结论
- 数据不足时主动降级或拒答
- 历史结论不能作为当前 live evidence

## 8步工作流
1. 确认持仓和时间跨度
2. Live fact gate - 拉取所有必需实时数据
3. 构建根因链 - 追溯到底层驱动因素
4. 市场体制分类 - risk-on/risk-off/event-compression/surprise-repricing
5. 资产排序 - BTC为方向锚
6. 输出唯一动作 - 11个枚举中选1个

8. 更新事件上下文

## 11因子分析体系
{factors_detail}

{decision_ladder}

## 事件压缩矩阵
{event_compression_matrix}

## 禁止事项
- 禁止模糊动作："wait for confirmation"、"watch key levels"、"be cautious"
- 禁止假设数据：如果数据缺失，标注缺失并降级
```

## 六、通知系统设计方案

### 6.1 通知渠道

| 渠道 | 优先级 | 触发条件 | 说明 |
|------|--------|----------|------|
| In-app Inbox | P0 | 所有 allowed + 用户配置的 blocked | 应用内通知收件箱 |
| Bark 推送 | P0 | allowed only | iOS 推送 |
| Web Push | P1 | allowed only | 浏览器推送（后续） |
| Email | P2 | 每日摘要 | 每日分析摘要（后续） |

### 6.2 通知与 HITL 的关系

```
分析完成
  ├── allowed（风控通过）
  │     ├── 用户在页面上 -> 展示分析卡片 + [确认][拒绝] 按钮
  │     │     ├── 用户确认 -> commit + 发送通知
  │     │     └── 用户拒绝 -> blocked(user_rejected)
  │     └── 用户不在页面上 -> 发送通知（含分析摘要）
  │           └── 用户从 Inbox 打开 -> 展示分析卡片 + [确认][拒绝]
  │
  ├── blocked（风控阻断）
  │     ├── notify_on_blocked=true -> Inbox 记录（不推 Bark）
  │     └── notify_on_blocked=false -> 仅数据库记录
  │
  └── no_trade
        ├── notify_on_no_trade=true -> Inbox 记录
        └── notify_on_no_trade=false -> 仅数据库记录
```

### 6.3 通知 Outbox 设计

通知发送通过事务 Outbox 模式：

1. `commit_final_artifact` 节点在同一事务中写入 `notification_attempts` 表（status=planned）
2. 独立 Outbox Worker 轮询 planned 状态的通知
3. Worker 获取租约（fencing token），发送通知，更新 status
4. 失败重试（有限次），最终 failed 进入 DLQ
5. 通知发送失败不改变已有决策或 Graph Run 终态

## 七、前端 UI/UX 设计方向

### 7.1 设计原则

1. **结论优先，细节按需展开**：用户首先看到方向/动作/价位，证据和推理按需展开
2. **流式渐进展示**：不要等全部完成才显示，每个阶段完成就更新 UI
3. **业务与诊断分离**：普通用户只看业务视图，诊断视图需要 `?mode=diagnostic`
4. **移动端优先**：考虑到用户可能在手机上查看通知后打开分析
5. **安全边界可见**：风控规则命中、数据缺失、manual_execution 必须清晰展示

### 7.2 页面设计概要

**Home 页面**：
- 市场简报卡片（BTC/ETH/SOL 最新价 + 24h 涨跌 + 资金费率方向）
- Watchlist 卡片列表
- 活跃任务列表（进行中的分析/研究/监控）
- 待处理事项（Inbox 未读数 + 最近待确认 Interrupt）
- 最近分析摘要（方向 + 结果 + outcome 状态）

**Work 页面（三栏）**：
- 左栏：Thread/Task 列表 + 新建按钮
- 中栏：对话时间线 + 阶段进度 + 分析结论卡片 + HITL 操作按钮
- 右栏：Artifact Inspector（行情快照/证据列表/风险门禁/模型详情）

**Inbox 页面**：
- 按类型分组：待确认 / 提醒 / 完成通知 / 失败恢复 / 配额告警
- 每条可展开详情 + 操作按钮
- 支持批量处理（如全部确认/全部忽略）

**Library 页面**：
- Artifact 列表（分析报告/研究报告/比较结果/复盘报告）
- 按标的/时间/方向/结果筛选
- Artifact 详情页：完整分析 + 证据 + 风控 + outcome

**Settings 页面**：
- 个人偏好（默认周期/通知渠道/风险参数）
- Watchlist 管理
- 持仓管理
- 通知配置
- 数据与隐私

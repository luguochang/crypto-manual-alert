# V2 重构方案评审与补充建议（修订版）

> 日期：2026-07-12
>
> 核心原则：一步到位做好完整设计，让 AI 按图施工。不渐进验证，不降低目标。
>
> 修订说明：本版撤销了上一版中"渐进式基础设施""Deep Agents 延后""Phase 0 拆分验证"等建议，改为聚焦设计文档的完备性和约束的精确性。

## 一、对重构思想的理解

### 1.1 为什么一步到位

在 AI 时代，技术实现难度不是瓶颈。瓶颈在于：架构设计是否完整、规范文档是否精确、约束文档是否能约束 AI 不走偏。如果设计文档做到位，AI 就能完全按照设计实现，不需要渐进验证。

V1 的根本问题不是技术选型错误，而是没有框架约束——AI（或开发者）在实现时为了"实现而实现"，自己造了一堆轮子：自建 workflow executor、agent swarm、orchestration runtime、tool executor、research pipeline、telemetry 系统和多套 store。这些在 LangChain/LangGraph 中都有官方实现。

### 1.2 V2 的核心思想

**最大化复用框架能力**。LangChain、LangGraph、Deep Agents 等框架已经封装好了：状态机、Tool 协议、Agent Loop、Structured Output、HITL、Streaming、Checkpoint、Thread/Run 生命周期、Middleware 等。V2 的业务代码只应该出现在：交易所行情标准化、风险规则、证据评估、产品 View Model 和业务表——其余全部用框架。

**AI 实现的最大误区**是为了实现而实现，不查官方文档，自己写了一堆框架已有的功能。因此约束文档必须精确到：每个需求对应哪个框架 API，禁止用哪种方式实现，CI 怎么检查。

### 1.3 评审聚焦点

本版评审不再质疑"是否应该一步到位"，而是聚焦：**现有设计文档是否完备到 AI 可以零决策实现**。具体检查：

1. 框架能力映射是否有缺口（某个需求没有指明用哪个官方 API）
2. 约束是否足够精确（AI 能否钻空子自己造轮子）
3. 官方文档引用是否充分（AI 实现时能否快速定位到正确 API）
4. 评测体系设计是否完整（不能只说"用 LangSmith Dataset"，要精确到指标定义、评测集结构、Judge Prompt）
5. 代码结构是否足够具体（每个文件的职责、接口、依赖关系是否明确）

## 二、现有 V2 方案评估

### 2.1 做得好的部分（应保留）

**02-official-framework-constraints.md 是整个 V2 的基石**。它把"优先使用官方框架"转化为了 CI 可执行规则，包括禁止模式列表、import-linter 契约、框架所有权测试和 schema 门禁。这直接解决了 V1 的根本问题。

**01-v2-product-and-architecture.md 的 Canonical Graph 设计**精确到了 10 个节点，每个节点明确了类型（确定性/Agent/Tool）和责任。这对 AI 实现非常友好。

**数据权威分离**（Checkpoint / Product DB / LangSmith / Langfuse）和**渐进持久化**设计是正确的。

**Wrapper Budget 一层规则**和**禁止模式列表**能有效防止 AI 造轮子。

### 2.2 需要补充的缺口

#### 缺口 1：框架能力映射不够精确到 API 级别

02 文档列了"必须优先使用"和"禁止替代实现"，但部分映射还停留在概念层面。例如：

- "Agent Loop → create_agent" 对，但没说 `create_agent` 的完整签名（model、tools、system_prompt、middleware、response_format、checkpointer 各怎么配）
- "HITL → interrupt / Command(resume=...)" 对，但没说在 Agent Server 环境下 checkpointer 由 Server 自动注入，代码中**不要再配**
- "Streaming → Agent Server protocol + @langchain/react" 对，但没说 `useStream` 的完整参数和 selector hooks 的具体用法

**建议**：补充一份"框架 API 精确映射表"（见第三节），每个需求精确到函数签名和官方文档链接。AI 实现时直接查表，不需要自己做架构决策。

#### 缺口 2：Agent Server 部署形态不够具体

01 文档选择了方案 A（LangGraph Agent Server + Next.js BFF），但 Agent Server 有三种部署模式（Cloud / Standalone Server / Self-hosted Full Platform），文档没有明确第一阶段选哪种。

从官方文档确认：**Standalone Server** 适合自有基础设施，用 Docker Compose 部署，需要自带 PostgreSQL + Redis，不需要控制面。这最适合本项目第一阶段。

**建议**：在 ADR-0008 中明确：第一阶段用 Standalone Server（Docker Compose），后续根据需求决定是否切到 Cloud 或 Self-hosted。

#### 缺口 3：Graph 编译方式未明确

官方文档明确推荐：**导出已编译的 CompiledGraph 实例**（推荐），而非 Factory function。因为 Server 在容器启动时加载一次，每个 Run 复用，无编译开销。Factory function 仅在需要 per-run 定制时使用。

官方还强调：**不要在图代码中配置 checkpointer 和 store**，因为 Agent Server 会在运行时自动注入。V2 文档没有明确这一点。

**建议**：在 01 文档的代码结构目标中补充：
- `graph.py` 导出 `graph = builder.compile()` 而非 `def build_graph(): ...`
- 代码中不配置 checkpointer/store（Agent Server 自动注入）
- `langgraph.json` 中注册编译后的 graph

#### 缺口 4：评测体系设计过于粗略

Phase 4 只提到"LangSmith Dataset、Experiment、Evaluator 和 Release Gate"，但缺少：
- 评测指标的具体定义和计算公式
- 评测集的数据结构和构建流程
- LLM-as-Judge 的 Judge Prompt 模板
- 回归测试的触发条件和门禁规则
- 轨迹归因的具体操作步骤
- Outcome 评测的成熟窗口定义和指标

**建议**：补充完整的评测体系设计文档（见第六节）。

#### 缺口 5：Custom Routes 设计不够具体

01 文档提到"Agent Server custom routes 提供产品业务查询和配置 API"，但没有说明：
- 哪些业务查询需要 custom routes（vs 哪些可以直接用 Agent Server Thread/Run API）
- Custom routes 的注册方式
- Custom routes 与 Graph 的关系（不能修改 Graph 内部 State）

**建议**：补充 custom routes 清单和注册方式。从官方文档看，custom routes 用 FastAPI APIRouter 注册在 `langgraph.json` 中。

#### 缺口 6：前端组件设计不够具体

01 文档列了页面路由和 UI 状态枚举，但缺少：
- 每个页面的 `useStream` 配置
- selector hooks 的具体使用方式
- custom channel 与 UI 组件的映射
- Generative UI 组件注册表的具体结构

**建议**：补充前端组件设计文档，精确到每个页面的数据源、selector 和组件树。

#### 缺口 7：V1 能力迁移矩阵缺失

V1 已验证的 OKX 行情获取、风控规则、Bark 通知等能力需要迁移到 V2，但没有明确的迁移矩阵。AI 实现时可能不知道哪些可以直接逻辑移植、哪些需要框架替换。

**建议**：补充 V1→V2 能力迁移矩阵（见第七节）。

### 5.1 HITL 完整工作流（含官方 API）

V2 文档第 12 节已设计了 HITL 场景，但需要精确到 API 调用：

**后端 - interrupt()**：
```python
# 在节点内调用
from langgraph.types import interrupt, Command

def confirm_analysis(state: AnalysisState) -> dict:
    # interrupt 前的代码必须幂等！resume 后会重新执行
    analysis = state["decision_draft"]

    # interrupt 暂停执行，value 传给前端
    approval = interrupt({
        "type": "analysis_confirmation",
        "data": {
            "direction": analysis.direction,
            "entry_range": analysis.entry_range,
            "stop_loss": analysis.stop_loss,
            "confidence": analysis.confidence,
        }
    })

    # resume 后，approval 是前端 respond() 传入的值
    if approval["action"] == "approve":
        return {"final_result": build_final(analysis, approved=True)}
    elif approval["action"] == "edit":
        edited = apply_edits(analysis, approval["edits"])
        return {"decision_draft": edited}
    elif approval["action"] == "reject":
        return {"final_result": blocked_result("user_rejected")}
```

**前端 - 检测和恢复**：

### 5.2 HITL 幂等性设计

官方文档的关键行为：**resume 后节点从头重新执行**。因此 interrupt 前的代码必须幂等。

V2 的 Canonical Graph 已经将外部调用拆成独立节点（`collect_market_snapshot`、`research_events`），interrupt 只出现在 `build_final_result` 之后。这个设计是正确的——interrupt 前的所有外部副作用已在独立节点完成并持久化。

补充约束：
```markdown
### HITL 幂等性约束

1. `interrupt()` 只能出现在没有外部副作用的节点中
2. interrupt 前的代码不能有：数据库写入、API 调用、随机数、当前时间读取
3. 如果 interrupt 需要展示数据，数据必须已在 State 中（来自前序节点）
4. 多个 interrupt 在同一节点内时，顺序不可随版本改变（resume 值按索引匹配）
5. 禁止 `while True + interrupt()` 作为通用输入校验器（用条件边循环）
```

```python
JUDGE_PROMPT = """
你是一个加密货币市场分析质量评审专家。请按照以下标准评估分析报告的质量。

## 分析报告
{analysis_output}

{reference_answer}

### 1. 方向准确性 (1-5分)
- 5分：方向完全正确，与参考答案一致
- 4分：方向基本正确，但置信度偏差较大
- 3分：方向模糊或部分正确
- 2分：方向错误但有合理推理
- 1分：方向完全错误

### 2. 风险控制 (1-5分)
- 5分：止损、入场区间、失效条件完整且合理
- 4分：大部分风控要素完整
- 3分：部分风控要素缺失
- 2分：风控要素严重不足
- 1分：无风控要素

### 3. 证据引用 (1-5分)
- 5分：引用了所有必需证据且推理逻辑清晰
- 4分：引用了大部分必需证据
- 3分：引用了部分证据但逻辑有跳跃
- 2分：证据引用不足
- 1分：无证据引用

### 4. 可执行性 (1-5分)
- 5分：manual_execution_required=true，参数完整可执行
- 4分：大部分参数完整
- 3分：部分参数缺失
- 2分：参数严重不足
- 1分：无法执行

"""
```

**校准流程**：

1. 建立 Golden Set（150-300 条人工标注样本），计算标注者间一致性（IAA）
2. 在 Golden Set 上运行 LLM Judge，计算与人工评分的 Spearman 相关和一致率
3. 分层抽样：高置信度样本抽检 5-10%，低置信度样本抽检 50-100%
4. 一致性目标：≥ 75%（人类之间也仅 65-80%）
5. 不一致案例用于迭代优化 Judge Prompt
6. 每月或模型更新时重新校准

### 6.5 轨迹归因设计

**归因流程**：

1. 对每个失败 case，在 LangSmith 中打开 Trace
2. 逐层展开 Trace，定位第一个出错的步骤
3. 按 Trace 节点类型归类问题环节

**Trace 节点与问题环节映射**：

| LangSmith Run Type | 对应环节 | 问题表现 | 优化方向 |
|---------------------|----------|----------|----------|
| `llm` (tool_calls) | 意图识别/路由 | 工具选择错误 | 优化 system prompt / tool description |
| `tool` (market) | 行情获取 | 数据缺失/过期 | 优化 provider / 超时配置 |
| `tool` (search) | 研究 | 搜索结果不相关 | 优化 search query / Tavily 配置 |
| `llm` (analysis) | 模型分析 | 方向错误/结构化输出不完整 | 优化 analysis prompt / few-shot |
| `chain` (risk_gate) | 风控 | 误拦截/漏拦截 | 调整风控规则参数 |
| `chain` (evidence_gate) | 证据评估 | 证据不足未阻断 | 调整证据阈值 |
| `llm` (final) | 最终生成 | 结果不完整 | 优化 final prompt |

**归因统计**：每次回归后生成失败原因分布图，一眼看出主要问题环节。

### 6.6 Outcome 评测设计

**成熟窗口**：分析生成后 4h/12h/24h（对应 horizon）

**采集方式**：Agent Server Cron Job 定期检查已到期分析，获取真实价格，计算 outcome

**指标**：

| 指标 | 定义 | 计算方式 |
|------|------|----------|
| Hit Rate | 方向判断正确率 | 正确数 / 总数 |
| Brier Score | 概率校准 | $\frac{1}{N}\sum(p_i - o_i)^2$ |
| MFE | 最大有利偏移 | 入场后最大盈利 |
| MAE | 最大不利偏移 | 入场后最大亏损 |
| PnL | 模拟盈亏 | 按入场/止损/止盈模拟 |
| No-Trade Baseline | 不交易基准收益 | 用于验证是否优于不交易 |

**分级门禁**（与 V2 文档 12 号一致）：

| 阶段 | 样本数 | 覆盖时间 | 门禁内容 |
|------|--------|----------|----------|
| 管道证明 | 1 | - | 证明 outcome 采集管道可用 |
| Beta | 50+ | 7天 | Hit Rate ≥ 55%, 无重大风控失误 |
| GA | 200+ | 30天 | Hit Rate ≥ 60%, Brier < 0.25, 风控 100% |

### 6.8 评测体系实施路线图

评测体系与 V2 Phase 同步推进，但不应等到 Phase 4 才开始：

## 八、补充建议

### 8.1 文档索引文件

建议在 `docs/v2/` 下创建一个 `OFFICIAL_DOC_INDEX.md`，将所有用到的官方文档链接集中索引。AI 实现时先查这个索引，不需要自己搜索。

### 8.2 langgraph.json 模板

建议在文档中给出 `langgraph.json` 的完整模板：

```json
{
  "dependencies": ["."],
  "graphs": {
    "agent": "./src/crypto_alert_v2/graph.py:graph"
  },
  "env": ".env",
  "http": {
    "app": "./src/crypto_alert_v2/api/routes.py:app"
  },
  "auth": {
    "path": "./src/crypto_alert_v2/api/auth.py:auth"
  }
}
```

### 8.3 前端 StreamProvider 模板

建议给出前端根布局的 `useStream` 配置模板，避免 AI 为每个页面单独配置。

### 8.4 评测 Judge Prompt 版本管理

Judge Prompt 应该和产品 Prompt 一样版本管理。建议在 LangSmith 中创建单独的 Dataset 存放 Judge Prompt 版本，每次校准时记录版本号。

## 九、总结

V2 的现有设计文档质量很高，框架约束纪律是整个方案的核心价值。本版建议聚焦于让设计文档**更加精确到 API 级别**，让 AI 实现时**零架构决策**：

1. **补充框架 API 精确映射表**（第三节）—— 每个需求精确到函数签名和官方文档链接
2. **加强 AI 实现者约束**（第四节）—— 明确禁止 AI 为了实现而实现的行为模式
3. **精确化 HITL 设计**（第五节）—— 含完整 API 调用示例和幂等性约束
4. **完整评测体系设计**（第六节）—— 含指标定义、评测集结构、Judge Prompt、回归门禁和轨迹归因
5. **V1 能力迁移矩阵**（第七节）—— 明确哪些逻辑移植、哪些框架替换、哪些完全废弃
6. **补充 Graph 编译约束**—— Agent Server 自动注入 checkpointer/store，代码中不配置
7. **补充 Custom Routes 清单**—— 明确哪些业务查询需要 custom routes
8. **评测体系尽早启动**—— Phase 1 就开始建评测集，不等 Phase 4
